### Фактический flow перед фазами нового user flow

```text
/start
  → билингвальное приветствие
  → выбор языка
  → текущая реализация сразу активирует trial и открывает главное меню
  → пользователь вручную выбирает «Поиск клиентов»
  → категория → направления → страна → вся страна/города → подтверждение
```

## 1. Аудит: найденные проблемы

### 🔴 Проблема 1: Два разных главных меню

В файле CLAUDE.md есть **два описания главного меню**:

**Экран 2 (основной):** 8 кнопок
```
🔍 Поиск клиентов
⚙️ Мои ключевые слова
📢 Мои каналы
📋 Мои подписки
🎁 Пригласить друга
💰 Тариф и оплата
🌐 Язык / Language
ℹ️ О сервисе
```

**Главное меню (обновлённое) в конце раздела UX:** 9 кнопок
```
🔍 Поиск клиентов
⚙️ Мои ключевые слова
📢 Мои каналы
📋 Каталог направлений  ← «Каталог направлений», а не «Мои подписки»!
🎁 Пригласить друга
💰 Тариф и оплата
🌐 Язык / Language
⚙️ Настройки             ← есть в обновлённом, нет в основном
ℹ️ О сервисе
```

**Различия:**
- «Мои подписки» vs «Каталог направлений» — разный нейминг
- В основном меню нет кнопки «⚙️ Настройки»
- В обновлённом меню кнопка «Настройки» есть

👉 **Нужно решение:** единое главное меню из 9 кнопок с единым неймингом.

---

### 🟡 Проблема 2: «🔍 Поиск клиентов» ведёт туда же, куда и «📋 Мои подписки → Подписаться»

Обе кнопки запускают одну и ту же FSM-воронку (выбор направления → страна → город). Это дублирование входа в одну и ту же функциональность.

👉 **Рекомендация:** Оставить кнопку «🔍 Поиск клиентов» как основной CTA (Call To Action) для входа в воронку. Кнопку «📋 Мои подписки» оставить только для управления существующими подписками (просмотр + отписка). Кнопку «Подписаться на направление» внутри «Мои подписки» — убрать, вместо неё текст «Чтобы добавить подписку, нажми 🔍 Поиск клиентов в главном меню».

---

### 🟡 Проблема 3: Onboarding wizard vs. FSM-воронка — разные тексты, одинаковая логика

Wizard шаги 1-2 и FSM шаги 9-12 делают одно и то же. Разница только в текстах и в том, что wizard заканчивается показом 10 запросов, а FSM — подтверждением подписки.

👉 **Рекомендация:** Использовать ОДИН код FSM для обоих сценариев, просто с флагом `is_onboarding=True`. Это сократит дублирование кода в 2 раза.

---

### 🟡 Проблема 4: Пустые состояния не описаны

Ни один экран не показывает, что видит пользователь когда:
- 0 подписок (экраны 5, 9)
- 0 ключевых слов (экран 3)
- 0 своих каналов (экран 4)
- 0 рефералов (экран 2a)
- 0 уведомлений сегодня (экран 2)
- Каталог пуст (нет стран/городов/сегментов)

👉 **Рекомендация:** Для каждого пустого состояния нужен текст-заглушка с призывом к действию. Например: «У тебя пока нет подписок. Нажми 🔍 Поиск клиентов чтобы найти первых!»

---

### 🟡 Проблема 5: Нет экрана ошибки оплаты

Описан только успех. Что видит пользователь при:
- Отклонённом платеже
- Таймауте платежа
- Недостаточно Stars

👉 **Рекомендация:** Добавить экран ошибки с кнопками «Повторить» и «Другой способ оплаты».

---

### 🟢 Проблема 6: «Назад» на разных экранах ведёт в разные места

На экране 3 (слова) кнопка «Назад» ведёт в главное меню [2]. На экране 5 (подписки) — тоже в [2]. На шаге 9 FSM — назад в [2] или в [5] в зависимости от того, откуда пришёл пользователь.

Это нормально, но в коде нужно хранить «предыдущий экран» в FSM-стейте.

---

### 🟢 Проблема 7: Нет экрана «Подтверждение удаления»

При нажатии 🗑️ на слове, канале, подписке — удаление происходит сразу? Или нужно подтверждение?

👉 **Рекомендация:** Показывать inline-подтверждение: «Точно удалить? [Да] [Нет]».

## 5. Периодические сообщения для Free-пользователей

Стратегия удержания и конверсии. Тон: benefit-oriented, без давления. Максимум 2 сообщения в неделю.

### 5.1 Календарь

| Тип | День | Время | Периодичность |
|---|---|---|---|
| 📊 **Итоги недели** | Понедельник | 10:00 | Каждую неделю |
| 🌱 **Новое в нише** | Четверг | 11:00 | Раз в 2 недели |
| 📈 **Твой месяц** | 1-е число | 10:00 | Раз в месяц |

При совпадении дат — приоритет: Твой месяц > Итоги недели > Новое в нише.

### 5.2 Условия отправки

- `plan = 'free'`, `is_blocked_bot = FALSE`, `is_banned = FALSE`
- Есть хотя бы одна активная подписка (user_subscriptions)
- Пользователь не отключил этот тип (periodic_prefs.is_disabled = FALSE)
- За период есть ненулевая активность в нише
- Не отправлялось в этом периоде (last_sent_at)

### 5.4 Модель данных

```sql
periodic_prefs (
    user_id     BIGINT REFERENCES users(id) ON DELETE CASCADE,
    msg_type    VARCHAR(30) NOT NULL CHECK (msg_type IN (
                    'weekly_digest', 'niche_growth', 'monthly_summary'
                )),
    is_disabled BOOLEAN DEFAULT false,
    last_sent_at TIMESTAMPTZ,
    PRIMARY KEY (user_id, msg_type)
);
```
