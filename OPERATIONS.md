## 1. Инцидент #1: FloodWait-бан (30.06.2026)

### Что произошло

При рефакторинге poller v2 был удалён rate limiter (`limiter.acquire()`).
195 каналов Hot-тира × 15 параллельных корутин × 2 API-вызова (get_entity + get_messages) =
~5850 запросов за 10 секунд → **18-часовой FloodWait от Telegram**.

### Корневая причина

Убрана защита от лавины запросов. `asyncio.gather(50 корутин)` без rate limiting =
мгновенный шторм на API Telegram.

### Хронология

| Время | Событие |
|---|---|
| 17:40 | Деплой poller v2 без `limiter.acquire()` |
| 17:42 | FloodWait 18h 27m от Telegram |
| 17:45 | Возвращён rate limiter (0.3 сек), PARALLEL_BATCH=3 |
| ~11:40 (+1д) | Ожидаемое снятие бана |

### Извлечённые уроки

1. **Rate limiter — не декорация. Это единственная защита от бана.**
2. Parallel batching безопасен только с rate limiting.
3. Перед любым изменением poller — считать ожидаемый RPS и сверять с лимитами.

## 1б. Инцидент #2: FloodWait-бан Account 2 (01.07.2026)

### Что произошло

При старте воркера все 3 тира (Hot/Warm/Cold) запускались **одновременно** через `asyncio.gather`.
Account 2 (новый, @Sofiya) получал 1018 каналов в первом цикле.
1018 × 2 API-вызова (get_entity + get_messages) = 2036 вызовов за ~11 минут непрерывного потока.

Даже при честных 3 rps, паттерн «новый аккаунт → чтение 1000+ каналов подряд» = триггер anti-spam.
Результат: **24-часовой FloodWait**.

### Корневая причина

Не скорость запросов, а **паттерн**: все тиры стартуют одновременно, создавая непрерывный поток API-вызовов
на протяжении многих минут. Telegram видит «новый аккаунт внезапно читает всё подряд».

### Извлечённые уроки

1. **Новые аккаунты — под особым надзором.** Нельзя сразу давать полную нагрузку.
2. **Тиры НЕ должны стартовать одновременно.** Разносить по времени.
3. **Нужен warmup-период** — постепенный выход на полную нагрузку.
4. **Предсказуемый паттерн = бан.** Нужен случайный jitter.

## 2. Hard Rules (нарушать запрещено)

### Rate Limiting

```
Правило #1: limiter.acquire() ДОЛЖЕН вызываться перед каждым API-запросом Telegram.
            get_entity() + get_messages() = 2 вызова limiter.acquire().

Правило #2: DEFAULT_MIN_INTERVAL ≥ 0.3 секунды на 1 аккаунт.
            3 запроса/сек = ~180 запросов/мин — безопасно.
            Telegram допускает ~30 запросов/сек, но мы держим запас ×10.

Правило #3: Перед изменением DEFAULT_MIN_INTERVAL или PARALLEL_BATCH —
            посчитать пиковый RPS: (каналов_в_тире ÷ PARALLEL_BATCH) × 2 acquire / интервал_цикла.
            Пиковый RPS НЕ должен превышать 5.
```

### Parallel Batching

```
Правило #4: PARALLEL_BATCH ≤ 5 на 1 аккаунт.
            Формула: PARALLEL_BATCH = min(5, max(1, floor(5 / кол-во_аккаунтов_в_тире))).
            С 3 аккаунтами: PARALLEL_BATCH ≤ 15 (по 5 на аккаунт).

Правило #5: BATCH_PAUSE ≥ 0.3 секунды между батчами.
            Даёт Telegram «выдохнуть» между пачками запросов.
```

### Circuit Breaker

```
Правило #6: limiter.wait_if_circuit_open() ДОЛЖЕН вызываться перед началом каждого батча
            и перед любым местом, где начинается массовый поллинг.

Правило #7: При получении FloodWait — НЕ ретраить, НЕ уменьшать интервал.
            circuit breaker открывается на seconds+10. Ждать.
```

### Tier Startup

```
Правило #10: Тиры НЕ должны стартовать одновременно.
            Hot=0s, Warm=60s, Cold=180s (STAGGER_STARTUP).
            Одновременный старт всех тиров создаёт непрерывный поток API-вызовов
            на 1000+ каналов → триггер anti-spam даже при честных 3 rps.

Правило #11 (обновлено 09.07.2026, задача B5): Warmup-рампа охвата УДАЛЕНА.
            Она была мертва с внедрения session-модели (skip_warmup при ≥1
            CB-free аккаунте) и, главное, рампа после каждого рестарта сама
            выглядит как калибровка бота. Плавность старта обеспечивают:
            последовательный опрос + лог-нормальные паузы (0.4), сессионные
            окна ACTIVE/PAUSED/SLEEPING (1.1), post-ban режим 48ч (2.2).
            Для НОВОГО аккаунта нагрузку наращивать вручную (мало каналов
            в маппинге → больше со временем), не кодом.

Правило #12: Jitter ±15% на интервалах. Предсказуемый паттерн = бан.
            Без jitter'а Telegram может засечь периодичность и принять за бота.
```

### Деплой изменений в poller

```
Правило #8: Перед деплоем — проверить логи за 5 минут после старта:
            docker compose logs -f worker | grep -E "FloodWait|circuit|error|Match"
            Если FloodWait в первую минуту — СТОП, откат.

Правило #9: Новый poller сначала тестируется на COLD-тире (15 мин интервал),
            затем на WARM, затем на HOT. Не наоборот.
```

## 3. Безопасные конфигурации

### 1 аккаунт (сейчас)

```python
DEFAULT_MIN_INTERVAL = 0.3   # 3 rps
PARALLEL_BATCH = 3           # 3 канала одновременно
BATCH_PAUSE = 0.3            # пауза между батчами

# Пиковый RPS: 3 канала × 2 acquire / 1.2 сек ≈ 5 rps (в пределах нормы)
# Hot-тир (195 каналов): 195 / 3 = 65 батчей × 1.2 сек ≈ 78 сек
```

### 3 аккаунта (будущее)

```python
DEFAULT_MIN_INTERVAL = 0.3   # на аккаунт
PARALLEL_BATCH = 5           # на аккаунт (5 × 3 = 15 всего)
BATCH_PAUSE = 0.3

# Hot-тир (195 каналов на 3 аккаунта = 65 на аккаунт):
#   65 / 5 = 13 батчей × 1.2 сек ≈ 16 сек
```

## 4. Процедура: что делать при FloodWait

### Шаг 1: не паниковать

Circuit breaker открылся, воркер ждёт. **Не перезапускать воркер вручную** — это сбросит таймер ожидания и вызовет повторный бан.

### Шаг 2: проверить масштаб

```bash
# Сколько осталось ждать?
CURRENT=$(date +%s)
EXPIRES=$(docker compose exec redis redis-cli GET circuit:expires_at)
echo "Осталось: $(( (EXPIRES - CURRENT) / 3600 )) часов"

# Кто виноват?
docker compose logs worker --tail=100 | grep "FloodWait\|circuit"
```

### Шаг 3: если бан > 1 часа — проверить аккаунт

```bash
# Залогиниться в Telegram с этого номера
# Проверить — может, спам-блок? Может, жалобы от пользователей?
# Если аккаунт не recoverable:
rm sessions/userbot.session
docker compose run --rm -it worker python -m app.userbot.auth
```

### Шаг 4: после снятия бана

```bash
# Убедиться что circuit breaker закрылся
docker compose exec redis redis-cli GET circuit:open
# Должен вернуть (nil)

# Проверить что поллинг возобновился
docker compose logs worker --tail=20 | grep "circuit breaker closed\|Match in @"
```

## 5. Чек-лист перед деплоем изменений в poller/rate-limiter

- [ ] `DEFAULT_MIN_INTERVAL` ≥ 0.3
- [ ] `PARALLEL_BATCH` ≤ 5 на аккаунт
- [ ] `limiter.acquire()` вызывается перед каждым API-запросом
- [ ] `limiter.wait_if_circuit_open()` вызывается перед началом батча
- [ ] Посчитан пиковый RPS: ≤ 5
- [ ] Посчитано время полного цикла для Hot-тира: ≤ 5 минут
- [ ] Плавность старта: последовательный опрос + лог-нормальные паузы + session-окна + post-ban (warmup-рампа удалена в B5, см. Правило #11)
- [ ] `STAGGER_STARTUP` присутствует (Hot=0s, Warm=60s, Cold=180s)
- [ ] `INTERVAL_JITTER` присутствует (±15%)
- [ ] Новый код протестирован на Cold-тире (15 мин интервал)
- [ ] `docker compose logs -f worker` — нет FloodWait/ошибок в первые 2 минуты

## 6. Известные уязвимости (актуально на 01.07.2026)

| Уязвимость | Риск | Статус |
|---|---|---|
| 1 аккаунт = single point of failure | При бане — полный простой | Ждём 2-3 SIM-карты |
| Нет мониторинга RPS в реальном времени | Бан обнаруживается постфактум | Не реализовано |
| Нет автоматического отката при FloodWait | Ручное вмешательство | Не реализовано |
| Circuit breaker сбрасывается при рестарте контейнера | Повторный бан после ребута | Redis persist, не критично |
