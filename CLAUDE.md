## 0. Правила разработки (обязательно к соблюдению)

- **Анализ до реализации** — Запрещено гадать. Если что-то неясно — задавай уточняющие вопросы. Озвучивай допущения и предлагай простые решения до написания кода.
- **Минимализм** — Код только для текущей задачи. Никаких заделов на будущее, если явно не попросили. Видишь способ сократить или написать проще — сокращай, пиши проще.
- **Точечные изменения** — Изменения только в целевом коде. Не трогать соседние функции и любую другую логику, если явно не попросили. Придерживайся зафиксированного стиля проекта. Подчищай за собой: если функционал больше не используется — удаляй. Удаляй только те зависимости, которые стали не нужны из-за твоих изменений.
- **Сложные задачи — композиция** — Для сложных задач пиши план: действие и метод проверки, с помощью которого будет валидироваться результат.
- **Маленькие шаги** — После каждого осмысленного изменения проект должен запускаться. Сломал — чиним прежде чем двигаться дальше.
- **Перед фазой показываю план файлов и жду подтверждения, потом пишу код.**
- **Каждая фаза = git commit + тег `phase-N-done` для точки отката.**
- **Стиль кода** — строго по `CODING_STYLE.md`. Нарушение конвенций = blocker на ревью.
- **Тестирование** — стратегия, пирамида тестов и QA checklist в `TESTING.md`. Обязательно с Фазы 2.
- **При поломке** — читай `RECOVERY.md` прежде чем что-либо делать.
- **Секреты только в .env** — никогда в код, никогда в git. Проверять `.gitignore` перед каждым коммитом.
- **Миграции БД всегда обратимы** — у каждой alembic-миграции рабочий `downgrade()`. Перед применением миграции — `pg_dump`.
- **Тесты обязательны** — С Фазы 2 пишем unit-тесты (pytest) параллельно с кодом. Правило: новая функция → новый тест. Дымовые/интеграционные тесты — после Фазы 5.

## 1. Что мы строим

SaaS-сервис на базе Telegram-бота, который отслеживает сообщения по ключевым словам в публичных каналах и чатах и присылает уведомления пользователям. Доступ предоставляется по платной подписке (freemium).

**Ключевая особенность:** для слежки за публичными пабликами, куда нельзя добавить бота, используется **userbot** (личный аккаунт через MTProto/Telethon), а Bot API используется только для отправки уведомлений и управления.

### Бизнес-логика монетизации

| | Free | Pro (~$5/мес) | Business (~$15/мес) |
|---|---|---|---|
| Матчинг | безлимитный (как у Business) | безлимитный | безлимитный |
| Подписки на направления | 1 | 3 | без лимита (кап: 60) |
| Свои каналы | 1 | 15 | без лимита (кап: 60) |
| Личных ключевых слов | 1 | 50 | без лимита (кап: 60) |
| Уведомлений/день | 50 | 150 | без лимита |
| Контакты в уведомлениях | СКРЫТЫ (@отправитель, чат) | Полные | Полные |
| Кнопки в уведомлении | «💰 Активировать подписку» | «💬 Ответить» (2 кнопки) | «💬 Ответить» (2 кнопки) |
| Regex-фильтры | — | да | да |
| Фильтр по языку | — | да | да |
| Просмотр каталога | да | да | да |
| Webhook / API | — | — | да |
| Дайджест уведомлений | да | да | да |
| Игнор-лист отправителей | да | да | да |
| CSV-экспорт заявок | — | — | да |
| Trial | — | — | **5 дней Business при первом прохождении воронки** |
| End-of-day отчёт | да (19:00 по часовому поясу) | — | — |

Скрытые капы Business (60 каналов, 60 ключей) — не показываются в UI, чисто защита от спама.

Лимиты раздельные: подписка на направление и свой канал считаются независимо. Channel-count направления НЕ влияет на стоимость (userbot подписан на канал 1 раз независимо от числа подписчиков); реальный контроль расхода/злоупотреблений — дневной лимит уведомлений. Числа лимитов — конфиг, меняются без правки кода.

### Trial-механика

- Новый пользователь при первом прохождении воронки получает **5 дней Business-тарифа** с полными контактами.
- При входе по реферальной ссылке: +3 дня к триалу (итого **8 дней** Business). На экране отображается: «5 + 3 бонусных дня».
- После истечения триала → понижение до Free: контакты скрыты, лимит 50 уведомлений/день.
- В 19:00 (по часовому поясу пользователя) — end-of-day отчёт: сколько заявок было всего, сколько получено. Контакты скрыты.
- При достижении лимита 50/50 — сообщение «Дневной лимит достигнут» с предложением подписки.
- Счётчики сбрасываются в 00:00.
- За 5 дней до окончания оплаченной подписки — авто-уведомление со статистикой.

### Реферальная программа

- **Двусторонний бонус:**
  - Referral (приглашённый): +3 дня к стандартным 5 дням триала (итого 8 дней Business).
  - Referrer (пригласивший): +7 дней к текущей подписке — **только когда referral оплатит Pro или Business**.
- Ограничение: не более 10 успешных рефералов в месяц (защита от abuse).
- Один пользователь может быть приглашён только один раз (`referral_id UNIQUE`).
- Механика: deep-link `https://t.me/LeadHunterBot?start=ref_<CODE>` → `/start` парсит код → запись в `referrals`.
- Бонусные дни добавляются к текущей подписке: если referrer на Free → получает +7 дней Pro; если уже на Pro/Business → подписка продлевается на +7 дней.

### Оплата

- **Старт:** Telegram Stars + CryptoBot (USDT). Интерфейс `PaymentProvider` — абстракция, конкретные реализации подключаются в Фазе 7.
- **Позже:** карты + QR СБП, YooKassa.
- Фазы 1–5 пишем БЕЗ оплаты (Free для всех), получаем рабочее ядро.

### Аудитория

Международная + русскоязычная. Следствия, закладываемые СРАЗУ с Фазы 1:
- **i18n обязателен** — тексты бота не хардкодим, храним в словарях `ru`/`en`, выбор по `language_code` пользователя Telegram.
- **discovery** — поиск каналов только на `ru` + `en`.
- Первый запуск: приветственный экран билингвальный (ru + en), с кнопками выбора языка. После выбора — весь интерфейс на выбранном языке.

### Каталог каналов v2 (курируется владельцем)

- **География двухуровневая:** `countries` → `cities`. Города не привязаны к темам — они отдельная ось классификации чатов.
- **Направления (segments)** — независимы от географии: работа, жильё, события и т.д. Один чат может входить в несколько направлений.
- **Чат (catalog_channels)** связан M:N и с направлениями (`channel_segments`), и с городами (`channel_cities`). Один чат может быть в нескольких направлениях и нескольких городах одновременно.
- **Редактирование каталога — ТОЛЬКО у владельца**, через веб-панель. У пользователей нет доступа к правке.
- **Навигация в боте:** выбор направления → выбор страны → «все города» ИЛИ выбор конкретных городов (галочки, множественный выбор) → подписка.
- **Наполнение** — через веб-панель на SQLAdmin (отдельный контейнер `admin`).

### Логика раздачи

Пользователю интересен чат X, если:
1. Чат X ∈ `channel_segments` для нужного направления
2. И (режим `all` ИЛИ чат X ∈ `channel_cities` для одного из выбранных городов)

Плюс пользователь может добавить **свои каналы** (ссылка / пересылка / `/discover`) — независимо от каталога.

### Алгоритм совпадения (семантическое ядро)

**Rule-based NLP-классификатор (трёхпроходный).** Без LLM на старте — чистая скорость, ноль затрат.

- **Поиск по границам целого слова**, регистронезависимо, с поддержкой Unicode/кириллицы (`(?<!\w)…(?!\w)`).
- **Подстроковый поиск НЕ используется:** «работа» НЕ матчит «работает»/«работал».
- **Трёхпроходный:** позитивный матч → минус-слова → структурные сигналы с приоритетами.
- **LLM-валидация отложена.** После тестирования rule-based классификатора на реальных данных — решение о подключении DeepSeek-V3 как второго этапа. Код `llm_validator.py` закладывается с Фазы 4, но вызывается только при `DEEPSEEK_ENABLED=true` в `.env`. По умолчанию — выключен.

```
│                                                                     │

        │  userbot слушает MTProto          │  Bot API шлёт уведомления
        ▼                                   ▼
   Публичные паблики                    Личные чаты пользователей
```

### Поток данных (per-message классификация)

```
userbot ловит NewMessage
    │
    ▼
classifier.classify(text) ──→ ["catering", "cleaning"]   ← 1 раз на сообщение!
    │
    ▼
find_interested_users(chat_username, matched_segment_ids)
    │  Redis-кэш sub:by_chat:{chat_username} → список user_ids
    │  + SQL для личных keywords
    ▼
Дедупликация: sha256(chat_username:message_id) + (user_id, message_hash) UNIQUE
    │
    ▼
Задача в очередь Redis (LPUSH queue:notifications)
    │
    ▼
sender.py: BRPOP + throttle 25/сек → Bot API → личка пользователю
```

### Per-user проверка (опциональная)

После per-message классификации, для каждого найденного пользователя проверяются **только личные keywords** (из таблицы `keywords`). Сегментные ключевые слова уже обработаны классификатором один раз.

```sql
users (

subscriptions (
    id              BIGSERIAL PRIMARY KEY,

    expires_at      TIMESTAMPTZ,
    payment_method  VARCHAR(20),                   -- stars / cryptobot / yookassa
    payment_status  VARCHAR(20) DEFAULT 'pending', -- pending / paid / failed / cancelled
    invoice_id      TEXT,                          -- ID инвойса в платёжной системе
    amount          DECIMAL(10,2),

)

-- ── КЛЮЧЕВЫЕ СЛОВА ПОЛЬЗОВАТЕЛЯ ──
keywords (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT REFERENCES users(id) ON DELETE CASCADE,
    text            TEXT NOT NULL,
    is_regex        BOOLEAN DEFAULT false,
    is_active       BOOLEAN DEFAULT true,
    created_at      TIMESTAMPTZ DEFAULT now()
)

-- ── СВОИ КАНАЛЫ ПОЛЬЗОВАТЕЛЯ ──
watched_chats (
    id                  BIGSERIAL PRIMARY KEY,
    user_id             BIGINT REFERENCES users(id) ON DELETE CASCADE,
    chat_username       VARCHAR(64),
    source              VARCHAR(20),              -- manual / discover
    userbot_account_id  INT,                      -- какой аккаунт следит
    title               TEXT,
    status              VARCHAR(20) DEFAULT 'approved',  -- approved / pending (приватный, ждёт вступления)
    is_private          BOOLEAN DEFAULT false,
    created_at          TIMESTAMPTZ DEFAULT now()
)

-- ── ДЕДУПЛИКАЦИЯ ОТПРАВЛЕННЫХ УВЕДОМЛЕНИЙ ──
sent_log (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT REFERENCES users(id) ON DELETE CASCADE,
    message_hash    VARCHAR(64),                  -- sha256(chat_username:message_id)
    is_urgent       BOOLEAN DEFAULT false,        -- 🔥 срочная заявка
    sent_at         TIMESTAMPTZ DEFAULT now(),
    UNIQUE (user_id, message_hash)
)

-- ── КАТАЛОГ v2 (курируется владельцем через веб-панель) ──

countries (                                       -- страна
    id          BIGSERIAL PRIMARY KEY,
    slug        VARCHAR(50) UNIQUE,               -- 'vietnam', 'indonesia'
    name_ru     TEXT,
    name_en     TEXT,
    is_active   BOOLEAN DEFAULT true
)

cities (                                          -- город внутри страны
    id          BIGSERIAL PRIMARY KEY,
    slug        VARCHAR(50) UNIQUE,               -- 'danang', 'bali'
    name_ru     TEXT,
    name_en     TEXT,
    country_id  BIGINT REFERENCES countries(id),
    is_active   BOOLEAN DEFAULT true
)

segments (                                        -- направление (независимо от гео)
    id          BIGSERIAL PRIMARY KEY,
    slug        VARCHAR(50) UNIQUE,               -- 'catering', 'housing', 'job-hiring'
    title_ru    TEXT,
    title_en    TEXT,
    emoji       VARCHAR(8),
    sort_order  INT DEFAULT 0,
    is_active   BOOLEAN DEFAULT true

segment_keywords (                                -- ключевые фразы направления

    text            TEXT NOT NULL,
    keyword_type    VARCHAR(20) DEFAULT 'demand', -- demand / stop / synonym
    is_regex        BOOLEAN DEFAULT false,
    is_active       BOOLEAN DEFAULT true,
    UNIQUE (segment_id, text, keyword_type)

catalog_channels (                                -- проверенный справочник чатов
    id              BIGSERIAL PRIMARY KEY,
    chat_username   VARCHAR(64) UNIQUE,
    title           TEXT,
    participants    INT,
    is_verified     BOOLEAN DEFAULT false,
    auto_matched_country_id BIGINT REFERENCES countries(id) ON DELETE SET NULL,
    auto_matched_city_id    BIGINT REFERENCES cities(id) ON DELETE SET NULL,
```

## 5а. Семантическое ядро (classifier.py)

### Архитектура: rule-based NLP (трёхпроходный)

```
Сообщение
  │
  ▼
classifier.py — трёхпроходный rule-based NLP
  │  Бесплатный, мгновенный, без внешних API
  │  Отсеивает оффер/болтовню через stop-фразы и структурные сигналы
  │  Возвращает список slug'ов категорий-спроса или []
  ▼
find_interested_users(chat, matched_segments)
```

**LLM-валидация (llm_validator.py) — опционально, на будущее:**
- Включается флагом `DEEPSEEK_ENABLED=true` в `.env`
- Добавляет второй этап после classifier: батч-запрос к DeepSeek-V3
- Стоимость: ~$0.40/мес при 200 каналах, $4/мес при 2000 каналах
- Код пишется с Фазы 4, но по умолчанию не вызывается
- Решение о включении — после сбора статистики false positives на реальных данных

### Алгоритм (трёхпроходный)

```
Шаг 1. ПОЗИТИВНЫЙ МАТЧ — хотя бы одна demand-фраза из сегмента?
        → НЕТ: игнорировать сообщение.
        → ДА: сообщение становится кандидатом.

Шаг 2. ФИЛЬТРАЦИЯ — прогнать кандидата через:
        - stop-фразы (segment_keywords с keyword_type='stop' + универсальные)
        - synonym-фразы (усиливают спрос, keyword_type='synonym')

Шаг 3. ПРИОРИТЕТ — структурные сигналы:
        - Если сильный глагол спроса стоит В НАЧАЛЕ фразы
          (ищу / нужен / нужна / нужны / сниму / возьму / посоветуйте /
           подскажите / хочу / где / кто делает) — он ПЕРЕБИВАЕТ
          слабые маркеры оффера. Оставляем.
        - В остальных случаях наличие маркера оффера ГАСИТ сообщение.
        - Вопросительный знак (?) усиливает «спрос».
        - Цена + контакт/ссылка/@username + хештеги усиливают «оффер».
```

### Структурные сигналы

**Признаки ОФФЕРА (понижают / гасят):**
- Цена + телефон/номер карты + ссылка/@username в одном сообщении
- 3+ хештега подряд
- Перечисление услуг списком с ценами

**Признаки СПРОСА (повышают):**
- Вопросительный знак
- Сообщение начинается с глагола спроса
- «кто-нибудь знает / подскажите / посоветуйте»

### Контекстно-зависимые правила

### LLM-валидация (опционально, на будущее)

**Модель:** DeepSeek-V3 (`deepseek-chat`), $0.07/1M input, $0.27/1M output.

**Промпт (батч-запрос):**

```
System: Ты классификатор. Отвечай ТОЛЬКО валидным JSON-массивом slug'ов.
        Если ни одна услуга не подходит — верни [].

User: Какие из этих услуг ищет автор сообщения?

      - cleaning: клининг, уборка

      Сообщение: «Ищу скутер напрокат на месяц, бюджет 200$»

Assistant: ["bike-rental"]
```

**Стоимость (расчёт):**

При 200 каналах: ~900 кандидатов/день → **$0.40/мес**.
При 2 000 каналах: ~9 000 кандидатов/день → **$4/мес**.

**Сигнатура:**

```python
# app/userbot/llm_validator.py

async def validate(
    text: str,
    candidate_slugs: list[str],
    segment_defs: dict[str, str]  # slug → описание из БД
) -> list[str]:
    """
    Отправляет ОДИН батч-запрос в DeepSeek со списком кандидатов (2-5 шт).
    Возвращает подтверждённые slug'и. Если ни один не подходит — [].
    """
    if not candidate_slugs:
        return []

    candidates_desc = "\n".join(
        f"- {s}: {segment_defs[s]}"
        for s in candidate_slugs if s in segment_defs
    )

    prompt = (
        "Какие из этих услуг ищет автор сообщения?\n"
        "Отвечай ТОЛЬКО JSON-массив slug'ов. Если ни одной — верни [].\n\n"
        f"Услуги:\n{candidates_desc}\n\n"
        f"Сообщение: «{text}»"
    )

    response = await deepseek_call(prompt)
    return parse_json_array(response)
```

## 5в. Retry-логика отправки уведомлений

| Ошибка | Действие |
|---|---|
| `403 Forbidden` | `users.is_blocked_bot = TRUE`, удалить из всех кэшей, больше не отправлять |
| `429 Too Many Requests` | `asyncio.sleep(retry_after)`, повторить |
| `5xx` / network error | До 3 ретраев: 1с, 4с, 9с. После 3 неудач → dead-letter очередь `dlq:notifications`, алерт владельцу |
| Throttle | 25 сообщений/сек (`asyncio.sleep(1/25)`) |
| Мониторинг очереди | `LLEN` на ключе очереди; алерт владельцу если backlog > 100 |

Dead-letter воркер: раз в час разбирает `dlq:notifications`, повторно пытается отправить.

```

  choosing_segment   → showing_countries →
  choosing_country   → choosing_mode     →
  choosing_mode      → choosing_cities / confirm_subscription →
```

- `/cancel` доступен всегда, сбрасывает state и данные
- Каждый шаг — inline keyboard с кнопкой «Назад»

```
Админ отвечает (AJAX POST /admin/chat/send)
    │
    ▼
bot.send_message(user_id, text) через Bot API
    │
    ▼
support_messages INSERT (direction='outgoing')
```

- WebSocket: `/admin/ws/chat/{user_id}` — подписка на сообщения конкретного диалога
- При загрузке страницы — AJAX GET истории сообщений (последние 50)
- Новые сообщения приходят через WebSocket, обновляют DOM без перезагрузки

## 7. Конвенции и правила работы

### Userbot — бережём

- Задержки 3–5 сек между вступлениями в каналы
- При `FloodWaitError` — ждём ровно `error.seconds`
- Лимит ~20–30 каналов на один userbot-аккаунт
- Для старта — **только публичные каналы**, без авто-вступления
- Закрытые каналы — позже, через ручное подтверждение в веб-панели
- Отдельный аккаунт для userbot — не основной личный номер

### Heartbeat-мониторинг

- Userbot пишет `last_heartbeat` в Redis каждые 15 минут
- Worker проверяет раз в минуту
- При пропуске >15 минут → алерт владельцу в личку через Bot API

### Бэкапы

- `pg_dump` раз в сутки в `/backups/`, ротация 7 дней
- Через cron на хосте (не в контейнере)
- Удалённое копирование — ручной процесс, автоматизируем позже

### Деплой

- Сервер = dev-окружение + продакшен
- `git push` с сервера на GitHub
- `docker compose up -d --build` для деплоя
- VSCode подключается через SSH для удалённого редактирования

### Веб-панель (admin) — доступ и безопасность

Первый запуск:
```bash
# Задать в .env:
#   ADMIN_PASSWORD=<надёжный пароль>
#   ADMIN_SECRET=$(openssl rand -hex 32)

docker compose up -d admin
docker compose logs admin --tail=5
```

Доступ с рабочего места (SSH-туннель):
```bash
ssh -L 8001:127.0.0.1:8001 user@server
# Открыть http://localhost:8001/admin/
```

Панель слушает `127.0.0.1:8001` на хосте — снаружи недоступна.

### Авторизация userbot (первый запуск / пересоздание сервера)

```bash
# 1. .env должен содержать API_ID, API_HASH, опционально USERBOT_PHONE
# 2. Интерактивный запуск:
docker compose run --rm -it worker python -m app.userbot.auth
# 3. Ввести телефон, SMS-код, пароль 2FA
# 4. Session-файл: ./sessions/userbot.session (в git не попадает)
# 5. При пересоздании сервера — скопировать sessions/userbot.session
```
