## GIT-СТРАТЕГИЯ: две версии проекта (решение владельца, 09.07.2026)

**Требование:** по завершении работ существуют две версии — «настоящая» (`main`, идентична текущему проду) и «после работ» (`audit/fable-fixes`). Обе тестируются перед окончательным переходом на проде.

1. **Все коммиты всех задач — ТОЛЬКО в ветку `audit/fable-fixes`** (создана от чистого main). Пометки о выполнении задач в этом файле коммитятся вместе с кодом задачи.
2. `main` не трогается до финального мержа — это эталон работающего прода и точка мгновенного отката.
3. **Промежуточные деплои в конце фаз ОТМЕНЯЮТСЯ** — вместо них: полный pytest в ветке + офлайн-верификация (eval-прогон корпуса из 0.1, golden-тесты эквивалентности). Прод продолжает работать с main.
4. **Два worker'а одновременно НЕ запускать** (main + ветка поллили бы Telegram одними userbot-аккаунтами → двойная нагрузка → бан, OPERATIONS.md hard rule). Сравнение версий — офлайн + контролируемое переключение.
5. **Переключение на новую версию (после завершения всех фаз):** pg_dump + бэкап сессий → прогнать `downgrade()` миграций ветки на копии → остановить worker → checkout `audit/fable-fixes` → build → up → чек-лист живой верификации (сценарии А/Б, HTML-лид, логи 15-30 мин, FloodWait, CPU).
6. **Откат (если новая версия ведёт себя плохо):** остановить worker → checkout `main` → build → up (секунды простоя). Redis-ключи совместимы между версиями (план не вводит новых форматов ключей).
7. **Финал:** после успешной живой верификации — merge `audit/fable-fixes` → `main`, тег `audit-fixes-live`.
8. ⚠️ Дисциплина: пока идут работы, любой `docker compose build` на сервере соберёт код ТЕКУЩЕЙ ветки — перед любой пересборкой проверять `git branch --show-current` (для прода — только `main` до финального перехода).

## ПРАВИЛА ДЛЯ ИСПОЛНИТЕЛЯ (обязательны, из CLAUDE.md §0 + OPERATIONS.md)

1. **Одна задача = одна ветка/коммит.** После каждой задачи проект запускается, тесты зелёные.
2. **Тест до фикса.** Для каждого бага сначала пишется падающий тест, воспроизводящий проблему, затем фикс делает его зелёным.
3. **Перед любой правкой** `poller.py`, `rate_limiter.py`, `classifier.py`, `pool.py` — прочитать OPERATIONS.md §2 (Hard Rules) и §5 (чек-лист).
4. **Прод не трогать при работающем worker.** Все задачи фазы разрабатываются и тестируются локально/в тестах; деплой — одним батчем в конце фазы: остановить worker → пересобрать → запустить → 2 минуты мониторить логи на FloodWait.
5. **Миграции** — только обратимые (`downgrade()`), перед накатом `pg_dump`. Писать вручную на хост + `docker compose cp` (техдолг №7: bind-mount миграций мёртв).
6. **Session log** — запись в CLAUDE.md §8 после каждой задачи + пометка в этом файле.
7. **После каждой фазы** — `/skill:phase-review`.
8. **Не расширять скоуп**: только описанное в задаче. Найденное попутно — фиксировать в этот файл (раздел «Найдено попутно» внизу), не чинить молча.

- `pool.initialize`: идти по маппингу, а не по `discover_sessions()`; session-файл отсутствует → аккаунт пропускается с warning. `discover_sessions` оставить только для диагностического лога «найдены неизвестные session-файлы».

**Тесты:** маппинг парсится; отсутствующий файл пропускается; порядок файлов в каталоге не влияет на ID (главный тест — добавить фиктивный `aaa.session`).

### [ ] B2. Производительность классификатора (вероятный источник CPU 100%)
**Контекст:** на каждое сообщение — до 1491 keyword × `re.escape`+`re.search` (thrashing кэша re на 512 паттернов) + лемматизация каждого keyword заново + universal stops × 69 сегментов.
**Что сделать (сохраняя поведение бит-в-бит):**
- При `_load_keywords` (poller) или в новом слое classifier'а: один раз предвычислить для каждого keyword — скомпилированный regex, лемма-форму, список слов, флаг multi-word. Структура: словарь slug → списки предвычисленных объектов. Universal stops — предкомпилировать отдельно, проверять ОДИН раз на сообщение (не в цикле сегментов), результат переиспользовать.
- `classify_message` принимает предвычисленную структуру (обратно-совместимая обёртка для тестов, принимающая сырые dict — компилирует на лету).
- Лемматизацию текста сообщения — оставить (один раз на сообщение, как сейчас); лемматизацию keywords из hot-path убрать полностью.
- Удалить `print(...)` (classifier.py:277).
- Кэш скомпилированных паттернов НЕ через модульный re-кэш, а явно в структуре.
**Верификация эквивалентности:** golden-тест — прогнать корпус из 0.1 (≥500 сообщений) старой и новой реализацией, результаты (matched_segments, is_urgent) идентичны на 100%.
**Бенчмарк:** микробенч 100 сообщений × полный keyword_map до/после, зафиксировать в session log (ожидание — ускорение ≥10×). После деплоя — наблюдать CPU worker.
