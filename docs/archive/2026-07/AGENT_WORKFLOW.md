# LeadHunter — как проводить работы и вести агента-кодера

> Составлено после разбора реального кода репозитория (`app/userbot/poller.py`, `rate_limiter.py`, `pool.py`, `classifier.py`, `config.py`, `docker-compose.yml`, `CLAUDE.md` и др.). Документ описывает **процесс**: как организовать репозиторий, как формулировать задачи агенту, в каком порядке идти и на что смотреть при приёмке. Список самих задач — в `LEADHUNTER_FIX_PLAN.md`; здесь — как их правильно исполнять с учётом того, что реально в коде.

---

## 0. Что важно знать до старта (найдено в коде, а не в аудите)

Аудит местами расходится с фактическим состоянием кода. Это первое, что нужно донести до агента, иначе он будет «чинить» уже сделанное или искать несуществующие файлы.

1. **Rate limiter — глобальный, не пер-аккаунтный.** `TelegramRateLimiter` в `rate_limiter.py` — синглтон с одним `self._last_call`. При двух аккаунтах он ограничивает *суммарный* темп, а не темп каждого аккаунта. То есть `DEFAULT_MIN_INTERVAL = 0.3` — это 3 rps на **оба** аккаунта вместе, и при этом внутри `_poll_batch` три корутины уходят через `asyncio.gather` почти одновременно. Это — реальный дефект, а не то, что описано в аудите. Первая же задача по поллеру должна сделать лимитер и бюджет **пер-аккаунтными**.

2. **Часть фиксов инцидента #3 уже в коде, часть — нет.**
   - `_distribute()` (poller.py:515) **уже** фильтрует заблокированные аккаунты до раздачи — этот фикс на месте.
   - Per-account try-lock (`_account_locks`, poller.py:485) **уже** есть.
   - Динамический Hot-интервал (`_get_effective_interval`, poller.py:557) **уже** есть (×2 при одном аккаунте).
   - **НО** `pool.handle_account_failure()` (pool.py:143) при этом **перераздаёт каналы упавшего аккаунта на живые** — прямо противоположно нужной деградации. Два механизма противоречат друг другу: `_distribute` уже не отдаёт каналы забаненному, а `handle_account_failure` их насильно перекидывает на выжившего. Это надо развести (см. Задача 0.2 плана).
   - Jitter внутри батча (poller.py:424) есть, но только ±30% на `BATCH_PAUSE` — а сам опрос всё ещё параллельный (`asyncio.gather`), что и есть машинный паттерн.

3. **`llm_validator.py` НЕ существует.** Аудит утверждает «код написан, выключен». В репозитории есть только ключи в `config.py` (`deepseek_api_key`, `deepseek_model`) и больше ничего. Задача 2.4 — это **написать с нуля**, а не «включить флаг».

4. **Кода фидбека (👍👎) нет вообще.** Задача 2.5 — greenfield.

5. **`get_entity` вызывается на каждый опрос** (poller.py:241). Telethon кэширует entity во внутреннем session-storage, поэтому это не всегда сетевой вызов — но паттерн хрупкий и при холодном кэше даёт лишний ResolveUsername. Проверить и сделать явный кэш (Задача 1.6).

6. **`report_flood_wait` ставит `seconds + 10`** (rate_limiter.py:61) — ровно то, что аудит называет причиной повторного бана между #1 и #3. Пост-бан режима нет (Задача 2.2).

7. **Бэкап сессий отсутствует.** `backup.sh` делает только `pg_dump`. `.gitignore` исключает `sessions/*.session` (правильно), но их никто не бэкапит наружу (Задача 0.7).

8. **У проекта уже есть зрелый рабочий процесс в `CLAUDE.md`.** Там прописаны: «анализ до реализации», «план файлов → подтверждение → код», git-тег `phase-N-done` после фазы, `/skill:phase-review`, обязательный session log в §8 после каждой задачи, и **hard rule**: перед любым изменением в `poller.py/rate_limiter.py/classifier.py/pool.py` читать §2 и §5 в `OPERATIONS.md` и 2 минуты мониторить логи после деплоя. **Не изобретайте новый процесс — работайте внутри этого.** Инструкции ниже расширяют его, а не заменяют.

## 1. Подготовка репозитория (один раз, до первой задачи)

Сделайте это сами, руками, до того как подпускать агента:

1. **Положите `LEADHUNTER_FIX_PLAN.md` и этот файл в корень репо** и добавьте ссылки на них в блок «Сопровождающие файлы» в шапке `CLAUDE.md`.

2. **Впишите инвариант бюджета в `CLAUDE.md §0` (Правила разработки):**
   > «Валюта системы — суточный бюджет API-запросов на аккаунт. Поллер строится от бюджета, не от списка каналов. Rate limiter и бюджет — пер-аккаунтные. Запрещено запускать worker против реальных аккаунтов без активного пер-аккаунтного лимитера и суточного бюджета.»
   Это ровно тот класс правил, что уже живёт в §0 — оно встанет естественно и агент прочитает его при старте каждой сессии.

3. **Ветка `main` защищена. Вся работа — в ветках `fix/<task-id>`** (напр. `fix/0.5-per-account-budget`). Одна задача = одна ветка = один PR.

4. **CI на GitHub Actions:** прогон `pytest` + линтер на каждый PR. `pytest.ini` уже настроен (`asyncio_mode = auto`, `testpaths = tests`). Минимальный workflow: поднять Python 3.11, `pip install -r requirements.txt`, `pytest`. Без CI ваши «критерии приёмки» держатся только на ручной проверке.

5. **Локальный способ запустить поллер в dry-run с моками** — понадобится для приёмки задач поллера. Если его нет, самой первой микрозадачей агенту сделайте тестовый харнесс: фейковый `UserbotAccount` (моки `get_entity`/`get_messages`), фейковый Redis (`fakeredis`), прогон одного цикла тира с логированием запросов. Без этого критерии Фазы 0 непроверяемы.

## 2. Главный принцип: одна задача — одна сессия — один PR

Ваш `CLAUDE.md` уже требует «план файлов → подтверждение → код». Усильте это для критичного кода:

**Никогда не давайте агенту «сделай Фазу 0».** Давайте ровно одну задачу из плана за сессию. Причины:
- Фаза 0 трогает `poller.py`/`rate_limiter.py`/`pool.py` — цена скрытой ошибки — четвёртый бан.
- Контекст агента ограничен; на одной задаче он держит только релевантный код.
- Каждый PR проходит критерии приёмки до следующего.

Цикл одной задачи (встроен в ваш процесс):
```
1. Вы: создать ветку fix/<task-id>, дать агенту промпт (шаблон §3).
2. Агент: прочитать CLAUDE.md §0, OPERATIONS.md §2/§5, LEADHUNTER_FIX_PLAN.md → нужную задачу.
3. Агент: НАЙТИ и показать релевантный код, предложить план файлов. СТОП.
4. Вы: подтвердить, что найдены правильные места.
5. Агент: сначала тест (критерии приёмки уже сформулированы как проверки), потом код.
6. Агент: pytest зелёный + dry-run поллера (для задач поллера).
7. Агент: дописать session log в CLAUDE.md §8.
8. Вы: ревью PR по критериям приёмки → merge. git-тег если это конец фазы.
```

Шаг 3 (найди-покажи-стоп) — не формальность. На этом коде агент легко «починит» глобальный лимитер, не заметив, что нужно делать его пер-аккаунтным, или тронет `handle_account_failure`, не увидев конфликт с `_distribute`. Вы проверяете, что он понял ландшафт, до единой строки правок.

СТОП. Жди моего подтверждения.

## 4. Красные линии для агента (добавить в CLAUDE.md §0)

- Не запускать `worker`/поллер против живых Telegram-аккаунтов. Живой прогон после снятия бана — только вы, вручную, на одном аккаунте, консервативно.
- Не коммитить `.env`, `sessions/*.session`.
- Не менять `docker-compose.yml` memory limits и деплой без отдельного явного запроса.
- Не «оптимизировать заодно» смежное — только scope задачи (это уже в §0 как «точечные изменения»).
- Любой инцидент с Telegram API — немедленно в `OPERATIONS.md` (это уже в §0).

## 5. Порядок работ с учётом реального кода

Следуйте фазам `LEADHUNTER_FIX_PLAN.md`, но с этими код-специфичными акцентами:

### Фаза 0 — по одной задаче, каждую принять полностью

Рекомендуемый порядок и на что смотреть:

- **0.5 (пер-аккаунтный бюджет) — ПЕРВОЙ, и расширить её.** В плане это «суточный бюджет», но по факту кода задача шире: `rate_limiter.py` надо сделать **пер-аккаунтным целиком** (свой `_last_call` и своё окно на каждый `account_id`, а не один общий). Бюджет `budget:used:{account_id}:{date}` — сверху. Без этого всё остальное в поллере стоит на песке. Приёмка: два аккаунта в dry-run не делят один лимит; исчерпание бюджета останавливает конкретный аккаунт.

- **0.4 (убрать параллельные батчи).** Заменить `asyncio.gather` по каналам одного аккаунта на последовательный обход; `BATCH_PAUSE`-jitter заменить на лог-нормальные паузы между запросами. `PARALLEL_BATCH` фактически убирается (или =1). Приёмка: в логах dry-run с одного аккаунта запросы строго последовательны, интервалы лог-нормальные.

- **0.2 (деградация) — здесь развести конфликт.** Убрать/переписать `pool.handle_account_failure()` так, чтобы каналы забаненного НЕ перекидывались на выжившего (сейчас перекидываются, pool.py:143). Оставить поведение `_distribute` (не отдавать каналы недоступному) + при одном живом аккаунте ставить Warm/Cold/Dormant на паузу. Приёмка: при одном живом аккаунте суммарная нагрузка ≤ бюджета одного, Warm/Cold/Dormant молчат.

- **0.3 (не поллить parked-страны).** Правка в `_rebuild_tiers` (poller.py:99): каналы стран без активной подписки → не в расписание. Сейчас они уходят в `dormant` и всё равно поллятся раз в 12ч. Приёмка: после ребилда в расписании 200–400 каналов вместо 2035, число parked в логе.

- **0.1** уже частично в коде — задача сводится к тестам на `_distribute` и try-lock (они есть, но без тестов) + проверке, что jitter/динамический интервал работают.

- **0.6 (Redis AOF)** — правка `docker-compose.yml` (том redis + `--appendonly yes`) — делаете скорее вы, чем агент (красная линия про compose). Агент готовит diff, применяете вы.

- **0.7 (бэкап сессий)** — расширить `backup.sh`: паковать `sessions/*.session` в шифрованный архив + выгрузка наружу. Обновить `RECOVERY.md`.

- **0.8 (CB при рестарте)** — при старте worker читать `circuit:expires:{id}` и не начинать поллинг до истечения; первый цикл после рестарта — на пониженном темпе.

### Фаза 1 — сессионная модель + наблюдаемость

- **1.1/1.2 (сессии, таймзоны)** — самая крупная переработка `_run_tier_loop`/`run_forever`. Непрерывный `while True` с `asyncio.gather` по тирам заменяется сессионной моделью (активные окна, паузы, ночной сон). Дайте это одной большой задачей, но с обязательным ЭТАПОМ 1 и dry-run симуляцией суток на приёмке.
- **1.3** уже есть в зачатке (`_get_effective_interval`) — расширить до 3 аккаунтов и вынести значения в конфиг.
- **1.4 (алерты)** — инфраструктура уже есть: `notify_admin()` работает, `heartbeat` пишется. Задача — добавить проверяющую таску (LLEN>100, 0 матчей за 3ч, бюджет исчерпан) + троттлинг алертов. Низкий риск.
- **1.6 (entity-кэш)** — проверить `get_entity` (poller.py:241), сделать явный кэш `channel_id/access_hash` per-account.
- **1.8 (дубли городов)** — Alembic-миграция, обратимая, `pg_dump` перед применением (правило §0).

### Фаза 2 — классификация (низкий риск бана, код изолирован)

- **2.4 (LLM) — писать с нуля**, файла нет. Отдельный `llm_validator.py`, fail-open, батчинг, кэш вердиктов в `class:cache`. Здесь агент безопасен — ошибка бьёт по качеству матчей, не по аккаунтам. Можно давать связанные задачи вместе.
- **2.6/2.7 (POS, offer-эвристики)** — правки в `classifier.py`, покрыты `test_classifier.py` (уже 7.5К тестов — расширять его).
- **2.5 (фидбек)** — greenfield: миграция таблицы + aiogram callback + вкладка админки.

### Фаза 3 — продукт (самое ценное при 3 пользователях, низкий риск)

Onboarding, рефералы, promo, fuzzy-города. Тексты RU/EN уже есть в `app/locales` и `USERFLOW.md`.

## 6. Приёмка: чему не верить на слово

Для задач поллера (0.1–0.5, 0.8, 1.1–1.3) зелёный `pytest` — необходимое, но НЕ достаточное. Перед merge сами прогоните dry-run и глазами проверьте в логах:
- запросы с одного аккаунта строго последовательны (нет одновременных);
- паузы лог-нормальные, а не константа;
- пер-аккаунтный бюджет соблюдается, исчерпание останавливает аккаунт;
- при одном живом аккаунте Warm/Cold/Dormant на паузе, каналы забаненного никуда не перекидываются;
- CB переживает симулированный рестарт.

Это ровно «Критерии выхода из Фазы 0/1» из плана.

**Живой прогон после снятия бана — только вы:** один аккаунт, консервативные значения (min_interval 1–2 сек, бюджет с большим запасом), 2 минуты мониторинга логов после старта (правило §0 CLAUDE.md), сутки наблюдения до подключения второго аккаунта.

## 7. Ежедневный ритм работы с агентом

1. Утром выбираете 1–3 задачи по порядку фаз.
2. По каждой — отдельная сессия, шаблон §3, обязательный ЭТАП 1 «найди-покажи-стоп».
3. Ревьюите PR по критериям приёмки из плана. Не мержите «на доверии» для кода поллера.
4. В конце фазы — `git tag phase-N-done` и `/skill:phase-review` (ваш существующий процесс).
5. Проверяете, что агент дописал session log в `CLAUDE.md §8` — без него задача не закрыта (ваше же правило).

## Резюме одним абзацем

У вас уже отличная дисциплина в `CLAUDE.md` — не ломайте её, а прицепите к ней три вещи: инвариант пер-аккаунтного бюджета в §0, правило «одна задача — одна сессия — один PR с обязательным этапом найди-покажи-стоп», и ручную приёмку dry-run для кода поллера. Первой задачей делайте пер-аккаунтный лимитер+бюджет (0.5, расширенную): текущий глобальный лимитер — корень того, почему anti-ban меры не спасали. И держите в голове три расхождения аудита с кодом: `llm_validator.py` не существует (писать с нуля), фиксы инцидента #3 частично уже в коде (проверять, не дублировать), а `handle_account_failure` активно вредит (перекидывает каналы на выжившего — развести с `_distribute`).
