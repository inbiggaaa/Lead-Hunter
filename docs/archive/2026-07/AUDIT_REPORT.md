# LeadHunter — Технический аудит (02.07.2026)

Документ для внешних специалистов. Содержит полное описание архитектуры, стека, инцидентов и открытых проблем.

---

## 1. Что такое LeadHunter

SaaS-сервис на базе Telegram-бота. Отслеживает сообщения по ключевым словам в публичных каналах и присылает уведомления пользователям. Freemium-модель: Free / Pro ($5/мес) / Business ($15/мес).

**Userbot** (Telethon/MTProto) слушает каналы, **Bot API** (aiogram) отправляет уведомления и управляет интерфейсом.

Аудитория: русскоязычная + международная (i18n: RU/EN).

## 5. Ядро поиска совпадений (classifier)

### 5.1 Трёхпроходный rule-based NLP

**Без LLM на старте.** `DEEPSEEK_ENABLED=false`. Классификация — 1 раз на сообщение, не per-user.

**Проход 1 — Demand:**
Поиск фраз по границе целого слова `(?<!\w)keyword(?!\w)`, Unicode, case-insensitive. 1935 demand-фраз распределены по 29 категориям. Нет demand → сообщение игнорируется.

**Проход 2 — Stop:**
Кандидат проверяется на 96 универсальных и сегментных stop-фраз. Stop-фраза без глагола спроса → гасим.

**Проход 3 — Структурные сигналы:**
Разрешение коллизий спрос/оффер:
- Глагол спроса в начале строки — перебивает оффер
- Цена + контакт/ссылка/@username + хештеги — усиливают оффер
- `?` больше не перебивает оффер-сигнал (только глагол спроса в начале)

**Срочность (🔥):** слова «срочно», «сегодня», «на завтра», «asap», «urgent» → `is_urgent=true`.

**Короткие anchors:** фразы с высоким риском ложного срабатывания («нужен байк», «хочу тату»). Матчатся ТОЛЬКО с контекстным сигналом спроса (глагол в начале, «?», «подскажите»).

**Channel pre-tagging:** каналы предтегированы сегментами по названию. Если канал pre-tagged + demand-сигнал → матч даже без keyword в тексте.

### 5.2 Грамматический анализатор

Используется pymorphy3 для лемматизации. **Проблема:** выбирает не ту часть речи для многозначных слов:
- «Наличные» (сущ.) → лемматизируется как «наличный» (прил.) → матчит «Наличный обмен» (реклама)

### 5.3 Offer-детектор

Регулярные выражения для определения коммерческих предложений. **Проблема:** не покрывает все форматы рекламы:
- EXASIA: нет цены/телефона — только «Добро пожаловать» → не детектится
- Kymco: цена без слова «цена» → не детектится
- Маркированные списки, цены в VND/донг, «Blue Card» → не покрыты

### 5.4 Отсутствует learning-цикл

Каждый false positive чинится вручную в коде. Нет механизма обратной связи:
- Нет кнопок 👍👎 в уведомлениях
- Нет авто-добавления стоп-слов на основе фидбека
- Нет A/B-тестирования правил классификации

### 5.5 Ключевые цифры classifier

- 1935 demand + 220 synonym + 102 stop = 2234 слов в БД
- 24 оффер-ориентированных синонима удалены
- Синонимы загружены для всех 29 сегментов
- 5 demand-сигналов добавлено: «требуется», «кто может», «подберите», «порекомендуйте», «есть у кого»
- 1 start-анкер: `^(где|куда|как|кто)\b.*\?`
- Убран паттерн `ищете/ищешь.*\?` — маркетинговый паттерн, не спрос

## 6. Механизм поллинга каналов (poller v2)

### 6.1 Архитектура поллера

**Инкрементальный:** Redis-курсоры `cursor:msg:{chat}` — только новые сообщения.

**Тирированный (4 тира):**

| Тир | Интервал | Критерий | Каналов |
|---|---|---|---|
| Hot | 60 сек | Каналы стран с активными подписками | 209 |
| Warm | 5 мин | Каналы > 1000 участников (ручные) | 0 |
| Cold | 15 мин | Каналы < 1000 участников (ручные) | 0 |
| Dormant | 12 часов | Всё остальное (неактивные страны) | 1827 |

**Параллельные батчи:** 3 канала × `asyncio.gather`, 0.3 сек между API-вызовами.

**Масштабирование:** `_distribute()` round-robin между N аккаунтами.

### 6.2 Rate Limiter

- `DEFAULT_MIN_INTERVAL = 0.3` сек (3 rps на аккаунт)
- `PARALLEL_BATCH = 3` на аккаунт
- `BATCH_PAUSE = 0.3` сек между батчами
- Circuit breaker: per-account, Redis-ключи `circuit:open:{id}`, `circuit:expires:{id}`
- FloodWait → circuit breaker открывается на `seconds + 10`

### 6.3 Anti-ban защиты (добавлены после инцидентов)

| Механизм | Описание |
|---|---|
| Staggered startup | Hot@0s, Warm@60s, Cold@180s |
| Warmup | 7 циклов: 8% → 16% → 25% → 35% → 50% → 70% → 100% |
| Jitter | ±15% случайной вариации на интервале цикла |
| Per-account CB | FloodWait одного аккаунта не блокирует остальные |
| Staggered tier starts | Тиры не стартуют одновременно |

## 7. Инциденты (FloodWait-баны Telegram)

### 7.1 Инцидент #1 — 30.06.2026 17:42 (18 часов)

**Причина:** Poller v2 запущен без `limiter.acquire()`. 195 каналов Hot-тира × 15 параллельных корутин × 2 API-вызова = **5850 запросов за 10 секунд**.

**Исправление:** Возвращён rate limiter (0.3 сек), PARALLEL_BATCH уменьшен с 50 до 3.

### 7.2 Инцидент #2 — 01.07.2026 10:30 (24 часа, Account 2)

**Причина:** Все 3 тира (Hot/Warm/Cold) стартовали **одновременно** через `asyncio.gather`. Новый аккаунт @Sofiya получил 1018 каналов в первом цикле → 2036 API-вызовов за 11 минут непрерывно. Telegram увидел «новый аккаунт внезапно читает всё подряд».

**Исправление:** Добавлены staggered startup, warmup, jitter.

### 7.3 Инцидент #3 — 02.07.2026 02:35 (10 часов, Account 1 — ПОВТОРНЫЙ)

Это самый сложный инцидент. Account 1 уже отбыл 18-часовой бан (#1), восстановился, и через 12.5 часов получил новый 10-часовой бан.

**Хронология:**

| Время (MSK) | Событие |
|---|---|
| 01.07 12:15 | Бан #1 истекает |
| 01.07 14:02 | Worker перезапущен, Account 1 начинает поллинг |
| 01.07 14:03 – 02.07 02:33 | **12.5 часов** непрерывного поллинга 105 каналов каждые 260 сек |
| 02.07 02:35:11 | Dormant-тир warmup step 2: **292 канала** (16% от 1827) |
| 02.07 02:35:12 | FloodWait 35956s — через **1 секунду** после старта Dormant |

**Корневая причина — три фактора:**

1. **Account 2 заблокирован → вся нагрузка на Account 1.** Функция `_distribute()` включает Account 2 в распределение (он healthy), но его чанк пропускается из-за circuit breaker. Каналы Account 2 **теряются**, а не перераспределяются на Account 1. При этом Account 1 тащит ВСЕ доступные каналы один.

2. **Dormant и Hot поллят один аккаунт одновременно.** Все 4 тира запущены через `asyncio.gather` и работают параллельно. Когда 12-часовой цикл Dormant совпал с активным Hot-циклом, оба тира начали долбить Account 1 одновременно.

3. **Предсказуемый паттерн.** 36 000+ API-вызовов за 12.5 часов с одинаковым ритмом (batch=3, pause=0.3, repeat) — Telegram засёк бота по регулярности. Jitter был только на интервале цикла, но не внутри батча.

**Исправления (реализованы 02.07, ожидают деплоя после снятия бана):**
- Per-account try-lock: тиры не поллят один аккаунт одновременно (skip, не starvation)
- `_distribute` фильтрует blocked-аккаунты ДО раздачи — каналы не теряются
- Динамический Hot-интервал: 120 сек при 1 доступном аккаунте (вместо 60)
- Jitter внутри `_poll_batch`: ±30% на BATCH_PAUSE

## 8. Открытые проблемы

### 🔴 Критические — надёжность

#### 8.1 Один живой аккаунт = одна точка отказа
Оба userbot-аккаунта периодически улетают в бан. Когда один забанен, второй тащит всю нагрузку и тоже улетает. Нужен 3-й аккаунт + ротация.
**Текущий статус:** ждём SIM-карту.

#### 8.2 Нет мониторинга
О проблемах узнаём только когда вручную смотрим логи. Нет алертов в Telegram:
- 0 матчей за час
- Падение воркера
- FloodWait > 1 часа
- Рост очереди уведомлений > 100

#### 8.3 Бэкап сессий
Если упадёт диск — userbot-аккаунты потеряны (сессии не бэкапятся вместе с БД).

#### 8.4 Circuit breaker сбрасывается при рестарте контейнера
Хотя CB пишется в Redis, рестарт воркера потенциально опасен — новый цикл warmup на только что разбаненном аккаунте.

---

### 🔴 Критические — классификация

#### 8.5 Лемматизация ломает матчинг
pymorphy3 выбирает не ту часть речи для многозначных слов:
- «Наличные» (сущ.) → «наличный» (прил.) → матчит «Наличный обмен» (реклама)

**Предлагаемое решение:** принудительный POS-тег для многозначных слов в keywords.

#### 8.6 Offer-детектор дырявый
Регулярками не покрыть все форматы рекламы:
- EXASIA: нет цены/телефона, только «Добро пожаловать»
- Kymco: цена без слова «цена»
- Маркированные списки, цены в VND/донг, «Blue Card»

**Предлагаемое решение:** эвристики по структуре сообщения (маркированные списки, цены в донгах, паттерны типа «Blue Card»).

#### 8.7 Нет learning-цикла
Каждый false positive чиним вручную в коде. Не собираем обратную связь от пользователей.

**Предлагаемое решение:** кнопки 👍👎 в уведомлениях → авто-добавление стоп-слов/минус-фраз.

### 🟡 Средние — продукт и рост

#### 8.8 3 пользователя — нет воронки привлечения
Реферальная программа написана но не проверена в бою. Нет promo-режима, демо-уведомлений.

#### 8.9 Скудный onboarding
Пользователь регистрируется и теряется. Нет wizard'а типа «Хочешь следить за клиентами в Дананге? Вот популярные сегменты → подпишись в 2 клика».

#### 8.10 Дубликаты городов в БД
«Самуи» (id=13 и 70), «🌐 Вся страна» (id=63) привязан к Турции.

#### 8.11 Старый discovery (v1) на 300-сек паузе
Discovery v1 всё ещё использует старый `DISCOVERY.md` без новых стран. Нужно объединить v1 и v2.

---

### 🟡 Средние — технический долг

#### 8.12 Dormant warmup растянут на 3.5 дня
Warmup Dormant-тира: 7 циклов × 12 часов = **84 часа до полной нагрузки**. Это означает что каналы из неактивных стран поллятся раз в 12 часов и только 8% из них в первые 12 часов. При добавлении нового пользователя из «неактивной» страны его каналы почти не поллятся 3.5 дня.

**Предлагаемое решение:** сократить Dormant warmup до 2-3 циклов, или форсировать ребилд тиров при изменении подписок.

#### 8.13 Нет LLM-валидации
Код `llm_validator.py` написан, но выключен (`DEEPSEEK_ENABLED=false`). При включении: один батч-запрос к DeepSeek-V3 после classifier — «Какие из [candidates] реально ищет автор?». Стоимость ~$0.40/мес при 200 каналах.

#### 8.14 Нет авто-теггинга городов с fuzzy matching
`_tag_new_channels()` ищет точное совпадение `city_name_ru in title_lower`. Но каналы пишут города с опечатками («Анталия» vs «Анталья»), транслитом («Alanya»), в падежах («Барахолка Нячанга»).

**Предлагаемое решение:** fuzzy matching (расстояние Левенштейна), поиск по name_en, нормализация окончаний.

#### 8.15 1165 каналов без привязки к городу
Привязаны только к стране. При mode='cities' — доставляются всё равно (city_id=NULL → фильтр пропускает). Это не баг, но создаёт шум.

#### 8.16 Качество discovery-поиска
Шаблон `{город} {слово}` не ловит обратный порядок («Барахолка Нячанга», «Общага Дананг»). 38-дневный цикл поиска — очень долго.

## 9. Redis-кэш и очереди

```
sub:by_chat:{chat_username}      → JSON [{user_id, segment_ids, keyword_texts, lang}]
class:cache:{message_hash}       → JSON ["slug1", "slug2"] (TTL 60 сек)
stats:daily:{uid}:{date}:matched → INT
stats:daily:{uid}:{date}:sent    → INT
limit_reached:{uid}:{date}       → "1" (TTL до полуночи)
queue:notifications              → LPUSH/BRPOP
dlq:notifications                → LPUSH/BRPOP
heartbeat:userbot:{id}           → timestamp
cursor:msg:{chat}                → INT (последний обработанный message_id, TTL 30 дней)
circuit:open:{id}                → "1"
circuit:expires:{id}             → unix timestamp
stats:unmatched                  → LPUSH (последние 10000)
stats:unmatched:seen             → SET (дедупликация, TTL 7 дней)
```

**Проблемы Redis:**
- Очередь — in-memory (LPUSH/BRPOP), при рестарте Redis уведомления теряются безвозвратно
- При росте нагрузки рекомендуется Redis Streams (XADD/XREADGROUP/XACK)
- Инвалидация кэша подписок — по триггеру + раз в час. Возможна задержка до часа

## 10. Retry-логика и обработка ошибок

| Ошибка | Действие |
|---|---|
| 403 Forbidden | is_blocked_bot=TRUE, удалить из кэша |
| 429 Too Many Requests | sleep(retry_after), повторить |
| FloodWait (MTProto) | Circuit breaker открывается на seconds+10, НЕ ретраить |
| 5xx / network | 3 ретрая (1с, 4с, 9с), затем dead-letter очередь |
| Throttle sender | 25/сек (sender_throttle_per_second) |
| Мониторинг очереди | LLEN, алерт при backlog > 100 |

**Проблема:** нет автоматического алерта — мониторинг описан но не реализован.

## 15. Альтернативные технологии и архитектурные решения

Мы открыты к пересмотру любых компонентов стека. Просим предложить альтернативные варианты по следующим направлениям:

### Язык и рантайм
- Оправдан ли Python (aiogram + Telethon) для этой задачи? Есть ли смысл переписать критичные компоненты (поллер, классификатор) на Go/Rust/Elixir?
- Какие плюсы/минусы у каждого варианта с точки зрения конкурентности, потребления памяти и экосистемы Telegram-библиотек?

### Хранение и очереди
- Redis LPUSH/BRPOP vs Redis Streams vs RabbitMQ vs NATS — что лучше для очереди уведомлений при росте до 10K+ пользователей?
- PostgreSQL vs ClickHouse для аналитики (статистика матчей, отчёты пользователей)?
- Нужен ли отдельный search-индекс (Elasticsearch/Meilisearch) для каталога каналов и поиска?

### Классификация
- Rule-based NLP (текущий) vs LLM (DeepSeek-V3/GPT-4o-mini) vs гибрид — что оптимально по соотношению точность/стоимость/задержка?
- Стоит ли смотреть в сторону embedding-моделей (text-embedding-3-small) + векторный поиск вместо keyword matching?
- pymorphy3 vs spaCy vs Natasha vs stanza — какой анализатор лучше для русского + вьетнамского + турецкого?

### Userbot и поллинг
- Telethon vs Pyrogram vs MtProtoKit (Go) vs прямой MTProto — что надёжнее для долгоживущих соединений?
- Есть ли альтернативы поллингу каналов? MTProto updates с подпиской на channel diffs?
- Как правильно реализовать ротацию аккаунтов и распределение нагрузки?
- Возможно ли использовать Telegram Premium-аккаунты для повышения лимитов?

### Инфраструктура
- Docker Compose на одном VPS vs Kubernetes (k3s) vs serverless — что оправдано на текущем масштабе и при росте?
- Нужен ли API Gateway (nginx/traefik) перед FastAPI?
- Какие инструменты мониторинга и алертинга (Grafana + Prometheus, BetterStack, Sentry-only)?

### Классификатор — NLP
- Есть ли готовые open-source решения для multilanguage intent detection, которые можно адаптировать?
- Стоит ли внедрять few-shot классификацию через LLM вместо rule-based?

**Мы не привязаны к текущему стеку.** Если предложенное решение лучше решает задачу — готовы мигрировать.
