**Ни одного tight-loop без sleep. Все циклы имеют корректные `await asyncio.sleep()`.**

### Корневая причина CPU 100%
CPU 100% — это **нормальное поведение** во время активной фазы поллинга. При одном CB-free аккаунте:
- Warmup ПРОПУСКАЕТСЯ (строка 540: `skip_warmup = (cb_free == 1)`)
- Все 217 Hot-каналов опрашиваются сразу
- Последовательный опрос с lognorm-паузами (медиана ~2с) длится ~400–500 сек на цикл
- В это время Telethon (криптография, сериализация MTProto) + classifier (regex на 2234 keyword) полностью загружают одно ядро

**Это НЕ баг.** При одном ядре VPS и Python asyncio это ожидаемо. После завершения батча цикл спит ~800с (1200с интервал − 400с работа).

### «Task destroyed but pending» для _alert_loop
Стек в логе: `Task-49 coro=<ChannelPoller._alert_loop() ...> wait_for=<Future ...>>`

Причина: при перезапуске worker (`docker compose restart`), event loop закрывается, и все pending таски уничтожаются. `_alert_loop` был в `await asyncio.sleep(300)` — т.е. ожидал sleep future. Это **безвредный артефакт shutdown** — graceful cleanup отсутствует, но данные не теряются (AOF сохраняет все Redis-ключи).

### Предлагаемая правка
**Ничего не делать.** CPU 100% — ожидаемо для 217 каналов на одно ядро. Shutdown-артефакт — косметика. Если нужно снизить CPU: увеличить `hot_interval_base` с 600 до 900 (→ 30 мин при 1 аккаунте), но это снизит частоту матчей.

Риск: 0. Рестарт: не нужен.

## D2. 32–48% ошибок Hot-тира (105 из 217)

### Что проверяли
- Все `except` блоки в цепочке поллинга: `_poll_channel` → `_poll_one` → `_run_tier_once`
- Точное тело `_poll_channel` (строки 283–418) и `_poll_batch` (строки 420–490)

### Найдено: ДВА слоя молчаливого глушения ошибок

**Слой 1 — `_poll_channel:300-302` (САМЫЙ КРИТИЧНЫЙ):**
```python
try:
    entity = await self._resolve_entity(account, channel_username)
except FloodWaitError:
    raise
except Exception:       # ← ChannelInvalidError, PeerIdInvalidError,
    return              #   ForbiddenError, TimeoutError — ВСЁ глушится БЕЗ лога
```

Этот `except Exception: return` ловит ВСЕ ошибки разрешения entity (ChannelInvalidError, PeerIdInvalidError, таймауты, PrivacyRestrictedError) и **молча пропускает канал без единой строки в лог**. При этом:
- `entity_cache` НЕ очищается (ChannelInvalidError должен удалять stale hash, но до очистки код не доходит — return раньше)
- Нет различия между «канал удалён» и «сетевой таймаут» — всё считается «error» в статистике батча

**Слой 2 — `_poll_one:465-466`:**
```python
except Exception as e:
    logger.debug("Poll error @%s: %s", ch.get('chat_username', '?'), e)
    return False
```

Этот ловит всё, что НЕ FloodWaitError и НЕ BudgetExceeded — включая ChannelInvalidError из `_fetch_all_since` и любые другие ошибки `get_messages`. Логируется на **DEBUG** (не видно в продакшене, где уровень INFO).

Также: `_fetch_all_since:415` имеет свой `except Exception: return []` — третий слой глушения для ошибок `get_messages`.

### Гипотеза о природе ошибок
105 ошибок из 217 = 48%. Возможные причины:
1. **ChannelInvalidError** — каналы удалены/приватизированы (в каталоге 2522, многие добавлены недели назад)
2. **PeerIdInvalidError** — username сменился
3. **ChatAdminRequiredError** / **ChannelPrivateError** — канал стал приватным
4. **Stale entity cache** — access_hash устарел (entity_cache не инвалидируется при ошибках в слое 1)

Точно узнать нельзя — все ошибки глушатся без лога.

### Предлагаемая правка (минимальная, обратимая)
В файле `app/userbot/poller.py`, строка 302 — заменить `return` на:
```python
except Exception as e:
    logger.warning("poll @%s: %s", channel_username, type(e).__name__)
    return
```

И строка 466 — заменить `logger.debug(...)` на:
```python
logger.warning("poll @%s: %s", ch.get('chat_username', '?'), type(e).__name__)
```

Это 2 строки. Позволит увидеть ТИПЫ ошибок (не полный текст) в логах уровня INFO/WARNING. Полностью обратимо — просто вернуть debug.

Риск: умеренный рост лог-объёма (105 строк раз в ~20 мин = ~7К строк/день). Не влияет на поллинг/бюджет/Telegram.
Рестарт: да (изменение в worker).

## D4. Post-ban не активирован

### Что проверяли
- Redis `KEYS "last_ban*"` и `KEYS "post_ban*"`
- Значения ключей и TTL
- Код `report_flood_wait()`, `activate_post_ban_if_recent()`, `_is_post_ban()`

### Найдено: ОШИБКА В АУДИТЕ. Post-ban АКТИВЕН.

```
last_ban_at:1     = 1783063537  (03.07.2026 10:25 MSK)
last_ban_at:2     = 1783054059  (03.07.2026 07:47 MSK)
post_ban_until:1  = 1783148785  (04.07.2026 10:06 MSK) — 18ч осталось
post_ban_until:2  = 1783148792  (04.07.2026 10:06 MSK) — 18ч осталось
```

Причина ложного вывода в аудите: команда `KEYS "post_ban:*"` использовала шаблон с двоеточием после `post_ban`, а реальные ключи имеют формат `post_ban_until:N` (нет ключа `post_ban:N`, есть `post_ban_until:N`). Redis KEYS с шаблоном `post_ban:*` не находит `post_ban_until:1` — нужно `post_ban*` (без двоеточия после префикса).

### Подтверждение работы post-ban
1. **Бюджет урезан:** `_is_post_ban()` → True → `effective_budget = 10000 // 2 = 5000`
   - acc1: 218/5000 (4.4%), acc2: 2516/5000 (50.3%) — лимиты соблюдаются
2. **Интервалы ×2:** `_get_effective_interval` → `max(degraded=2.0, post_ban=1.5) = 2.0`
   - Hot = 600 × 2.0 = 1200с (20 мин) — верно для 1 аккаунта
   - post_ban ×1.5 перекрывается degradation ×2.0 — это корректно (max, не умножение)
3. **Активация при старте:** `start()` → `activate_post_ban_if_recent()` проверил `last_ban_at` → бан был <48ч назад → установил `post_ban_until`

### Предлагаемая правка
**Ничего не делать.** Post-ban работает корректно. Исправить неточность в AUDIT_STATE.
