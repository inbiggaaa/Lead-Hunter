# U0 — функциональный контракт базового user flow

Дата: 14.07.2026. Статус: принят как консервативный контракт реализации. Он не добавляет маркетинговых обещаний и не меняет lifecycle до специальных фаз.

## Продуктовые решения

1. RU-интерфейс последовательно обращается на «ты»; EN использует нейтральный friendly product English.
2. Пользовательский термин: «Мои поиски» / “My searches”. «Подписка» означает только оплаченный доступ.
3. Целевое главное меню: «Мои поиски» или «Настроить первый поиск», «Заявки и результаты», «Тариф», «Пригласить друга», «Настройки». Доступность результата зависит от тарифа; пустые состояния ведут к настройке поиска.
4. Trial активируется только после успешного создания первого поиска. Referral увеличивает будущий trial до 8 дней, но не запускает его до создания поиска.
5. При 0 лидах за trial стандартные сообщения упущенной выгоды запрещены. Базовый путь: диагностика поиска → расширение географии/фраз/каналов → поддержка. Автоматическое продление trial не обещается и не реализуется без отдельного решения.
6. Текущая техническая grace-модель paid: 7 полных дней после `plan_expires_at`, затем downgrade в Free с сохранением поисков. Тексты lifecycle и стимулы продления будут отдельно спроектированы после базового userflow; до этого нельзя говорить, что доступ уже закрыт, пока plan ещё paid.
7. Free-лид: текст заявки, название чата plain text и совпавшие направления; автор, ссылки на чат/сообщение и кнопки контакта скрыты. Регион и время сейчас не показываются и не обещаются. Paid/Trial получают доступные ссылки на чат и автора, если payload позволяет их построить.
8. Возврат к конкретному скрытому лиду после оплаты технически не реализован. В базовом flow это не обещается; отдельная задача допустима после появления персистентного lead reference.
9. До отдельного маркетингового этапа новые proactive-сообщения не добавляются. Текущий scheduler считается baseline, а не утверждённой частотой. После базового userflow отдельно проектируются тексты, expiry/renewal и повторная подписка.
10. Льгота за раннее renewal и comeback-бонус отсутствуют. Не обещать до отдельного бизнес-решения и реализации в billing.

## Promise-to-function audit

| Обещание / расхождение | Факт runtime | Решение |
|---|---|---|
| Onboarding ведёт к созданию поиска | после `lang:*` сразу menu | реализовать единый onboarding через catalog FSM в U3 |
| Trial после получения ценности | стартует после языка; referral может стартовать в `/start` | перенести активацию после commit первого поиска в U3 |
| Сквозной выбранный язык | sender/reminders и часть handlers RU-hardcoded | реализовать i18n-контракт в U1; до этого не считать EN завершённым |
| Paid grace и текст expiry | план остаётся paid 7 дней, downgrade позже | формализовать тексты в U8; не писать «контакты скрыты» до фактического downgrade |
| Business «без лимитов» | caps 12/9/unlimited cities/50/50 | использовать точные caps; карточки v2.1 уже исправлены |
| Доставка «за 2 секунды» | polling, latency варьируется | удалить обещание; использовать только измеренную latency |
| «Давно не виделись» | query использует `created_at` | не отправлять как inactivity после переработки; добавить `last_active_at` в U8/U9 |
| Onboarding обещает заявки | legacy placeholder, поиска нет | удалить placeholder; показывать успех только после созданного поиска |
| EN parity | смешанный язык в основных путях | U1 blocker для следующих UX-фаз |
| «Найдено N» | menu — Redis matched; stats — sent_log; EOD — daily counters | явно называть matched/sent; унифицировать продуктовую метрику в U2/U7 |
| Вернуться к скрытому лиду после оплаты | lead reference не хранится для UI return | не обещать; отдельная функциональная задача |
| «Поиск запущен» | истинно только после Subscription commit и cache invalidation | экран успеха показывать строго после обоих действий |
| «Ответить одним нажатием» | возможно не для каждого private/numeric payload | показывать CTA только при валидной ссылке |
| «Новые каналы добавлены» | periodic message не проверяет факт добавления | удалить/переписать в маркетинговой фазе |

## Baseline аналитики

| Этап | Наблюдаемость сейчас | Контракт U2 |
|---|---|---|
| `/start`, выбор языка | отдельного события нет | `onboarding_started`, `language_selected` |
| начало/завершение поиска | отдельного события нет | `search_setup_started`, `search_created`, `search_setup_failed` |
| trial activation | поля User + admin notification | `trial_activated` с source и duration |
| matched/delivered lead | Redis counters + sent_log | `lead_matched`, `lead_delivered`, единый correlation id |
| feedback | Feedback в DB | `lead_feedback_submitted` |
| paywall | Redis `stats:paywall:{trigger}` | сохранить trigger, добавить user/plan/source context |
| plan/period/provider/invoice | Payment + callbacks, этапы неполны | события выбора и `invoice_created/failed` |
| payment success/failure | Payment/status, provider branches | единая схема `payment_succeeded/failed` |
| renewal | можно вывести из Payment | явный `subscription_renewed` |
| expiry/downgrade/winback | Reminder + User/Payment | lifecycle events и conversion attribution |

Числовой baseline конверсий сейчас корректно восстановить нельзя: нужные события отсутствуют. До U2 запрещено придумывать исходные проценты; текущие Redis paywall counters годятся только как частичная техническая метрика.

## Gate U0

- Тарифная матрица v2.1 подтверждена runtime-тестами; дневного лимита уведомлений нет.
- Полный реестр поверхностей: `docs/userflow_screen_registry.md`.
- Все известные обещания получили статус «подтверждено», «реализовать» или «не обещать».
- Маркетинговые тексты, expiry и механики повторной подписки намеренно вынесены после базового userflow по решению владельца от 14.07.2026.
