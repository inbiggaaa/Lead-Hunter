```python
async def _fetch_all_since(
    self, account, entity, channel_username: str, cursor: int,
) -> list:
    """Fetch messages since cursor. Single API call, no pagination."""
    await limiter.acquire(account.account_id)
    try:
        if cursor > 0:
            batch = await account.get_messages(entity, min_id=cursor, limit=FETCH_LIMIT)
        else:
            batch = await account.get_messages(entity, limit=FETCH_LIMIT)
        return list(batch) if batch else []
```

## 2. КУРСОР — критично

### Чтение: `_get_cursor()` — line 239–248

```python
@staticmethod
async def _get_cursor(chat_username: str) -> int:
    """Get last processed message_id for a channel (0 if new)."""
    try:
        from app.cache import get_redis
        redis = await get_redis()
        val = await redis.get(f"{CURSOR_PREFIX}{chat_username}")
        await redis.aclose()
        return int(val) if val else 0
```

### Запись: `_set_cursor()` — line 251–262

```python
@staticmethod
async def _set_cursor(chat_username: str, msg_id: int) -> None:
    """Update cursor to the latest processed message_id."""
    try:
        redis = await get_redis()
        await redis.set(f"{CURSOR_PREFIX}{chat_username}", str(msg_id))
        await redis.expire(f"{CURSOR_PREFIX}{chat_username}", 30 * 86400)
        await redis.aclose()
```

**Ключ:** `cursor:msg:{chat_username}` в Redis, TTL 30 дней.

### Как вычисляется max_msg_id — line 343–410

```python
max_msg_id = cursor                          # line 344
for msg in reversed(all_messages):
    if not isinstance(msg, Message) or not msg.message:
        continue                             # ← SKIP без обновления max_msg_id!
    msg_id = msg.id
    if msg_id > max_msg_id:                  # line 348
        max_msg_id = msg_id                  # line 349
    # ... classify, dispatch, log ...
# Update cursor to latest processed message
if max_msg_id > cursor:                      # line 409
    await self._set_cursor(channel_username, max_msg_id)  # line 410
```

**ВЫВОД (однозначно):** Новый курсор берётся из **УЖЕ ОБРАБОТАННЫХ/ОТФИЛЬТРОВАННЫХ** сообщений — тех, что прошли проверку `isinstance(msg, Message) and msg.message`. Сообщения без текста (фото, стикеры, служебные) пропускаются через `continue` ДО обновления `max_msg_id`, поэтому их id НЕ попадают в курсор.

**Последствия:** если в батче последние сообщения — без текста (например, фото без подписи), их id не войдёт в курсор → на следующем опросе они перечитаются заново (но min_id отфильтрует их серверно, т.к. они ≤ старого курсора? НЕТ — они НОВЕЕ курсора, ведь max_msg_id не был обновлён). Это баг: курсор отстаёт от реального состояния канала → лишние API-вызовы.

## 6. Поведение первого прохода (cursor=0)

При `cursor == 0` (новый канал или сброшенный курсор):
```python
batch = await account.get_messages(entity, limit=FETCH_LIMIT)  # без min_id
```
Telegram возвращает ПОСЛЕДНИЕ `FETCH_LIMIT=100` сообщений канала. Нет ограничения по дате — если в канале 10 сообщений за всё время, вернутся все 10. Если 10000 — только последние 100. Затем курсор выставляется на `max_msg_id` обработанных сообщений, и следующие опросы используют `min_id`.

## МИНИМАЛЬНЫЙ ПЛАН ПРАВКИ

### Шаг 0: конфиг
В `app/config.py`, класс `Settings`, добавить:
```python
message_max_age_days: int = 7  # skip messages older than N days
```
Не хардкод-магия, меняется через `.env`.

### Шаг 1: вычисление cutoff
В `_poll_channel()`, после получения `all_messages` и перед циклом обработки:
```python
from datetime import datetime, timezone, timedelta
cutoff = datetime.now(timezone.utc) - timedelta(days=settings.message_max_age_days)
# cutoff — tz-aware UTC
```

### Шаг 2: фильтр по дате В ЦИКЛЕ
Внутри `for msg in reversed(all_messages):`, сразу после проверки `isinstance(msg, Message) or not msg.message`, добавить:
```python
if msg.date and msg.date < cutoff:
    msg_id = msg.id                           # обновить max_msg_id!
    if msg_id > max_msg_id:
        max_msg_id = msg_id
    continue                                  # пропустить старые сообщения
```

**Критично:** `max_msg_id` обновляется ДО `continue` — чтобы курсор продвинулся за пределы старого сообщения, иначе оно будет перечитываться бесконечно.

### Шаг 3: курсор — по ПОЛНОЙ серверной выдаче
Текущий код НЕ продвигает курсор по полной выдаче (пропускает сообщения без текста). Это pre-existing баг, выходящий за scope задачи. В рамках ЭТОЙ задачи — только обеспечить, что старые сообщения не блокируют курсор (см. шаг 2 выше).

**Если бы мы чинили курсор до полной выдачи (scope creep, НЕ ДЕЛАТЬ сейчас):** — перенести `msg_id = msg.id` и обновление `max_msg_id` до всех фильтров:
```python
for msg in reversed(all_messages):
    if not isinstance(msg, Message):
        continue
    msg_id = msg.id
    if msg_id > max_msg_id:
        max_msg_id = msg_id
    if not msg.message:
        continue
    if msg.date and msg.date < cutoff:
        continue
    # ... classify, dispatch ...
```

### Шаг 4: осознанный компромисс
При разрыве >7 дней между опросами (например, канал не опрашивался неделю из-за бана аккаунтов), сообщения «из серёдки» (новее курсора, но старше cutoff) пропускаются **намеренно**. Они не попадают ни в лиды, ни в unmatched-логи. Это trade-off: свежесть > полнота. Альтернатива (обрабатывать все N сообщений за неделю) создала бы лавину старых лидов при восстановлении после бана.

### Шаг 5: первый проход (cursor=0)
При cursor=0 Telegram возвращает последние 100 сообщений. Фильтр по дате отсечёт те из них, что старше 7 дней. В активных каналах последние 100 сообщений обычно ≤ 1–2 дня, поэтому фильтр почти никогда не сработает. В малоактивных каналах (1 сообщение в неделю) все сообщения пройдут. Всё ОК — `FETCH_LIMIT=100` + датовый пол достаточны.

### Шаг 6: тесты (после implement)
- unit: сообщение старше 7 дней → `continue` (не классифицируется, не диспатчится)
- unit: курсор продвигается по `server_max`, даже если весь батч старый и 0 dispatch
- unit: безтекстовое сообщение в хвосте батча → курсор всё равно уходит вперёд (pre-existing bug fix)
- unit: сообщение младше 7 дней → нормальный путь (classify → dispatch)
