## Ключевая архитектурная директива (прочитать до начала работы)

Вступление в каналы (join) **недоступно** — работаем только через `GetHistory` по каналам, где аккаунт не состоит. Из этого следует главный инвариант всей системы:

**Валюта системы — суточный бюджет API-запросов на аккаунт, а не список каналов.** Поллер строится сверху вниз от бюджета (~8–12K запросов/сутки на выдержанный аккаунт), а не снизу вверх от «сколько каналов надо опросить». Telegram банит не за RPS, а за регулярность и непрерывность паттерна. Отсюда:

- Hot-интервал 60 сек при 2 аккаунтах физически невозможен. Целевой SLA — «лид за 15–30 минут», не real-time.
- Последовательный опрос вместо параллельных батчей.
- Сессионная модель с длинными паузами и ночным сном вместо непрерывного цикла.
- Не поллить каналы стран без подписчиков вообще.

Все задачи ниже реализуют этот принцип.

# ФАЗА 0 — Стабилизация (до снятия текущего бана)

Цель фазы: устранить причины повторных банов и потерь данных. Ничего из этого не требует живого аккаунта — можно и нужно делать пока аккаунт в бане.

## Задача 0.1 — Покрыть тестами уже реализованные фиксы инцидента #3

**Контекст.** Вопреки формулировке аудита «ждут деплоя», фиксы **уже в коде**, но без тестов. Реализовывать заново НЕ нужно — только покрыть тестами и проверить корректность. Что уже есть:
- Per-account try-lock (`_account_locks`, poller.py:485): при занятости аккаунта другим тиром — `skip`.
- `_distribute()` (poller.py:515) фильтрует заблокированные аккаунты (`is_healthy` + `is_circuit_open`) **до** раздачи.
- Динамический Hot-интервал (`_get_effective_interval`, poller.py:557): ×2 при 1 аккаунте.
- Jitter между батчами (`_poll_batch`, poller.py:424): ±30% на `BATCH_PAUSE`.

**Что сделать.**
1. Прочитать перечисленные функции, подтвердить их поведение (не менять логику в рамках этой задачи).
2. Написать unit-тест на `_distribute()`: при 1 blocked из 2 аккаунтов каналы не теряются и не уходят заблокированному.
3. Написать unit-тест на try-lock: если аккаунт занят (lock.locked()), второй тир получает skip, а не ждёт.
4. Написать unit-тест на `_get_effective_interval`: 1 аккаунт → base×2, 2+ → base.

**Приёмка.** Тесты зелёные. Ручная трассировка dry-run: «Account 2 blocked» → Account 1 не получает каналы Account 2 через `_distribute`. (Отдельно: конфликт с `handle_account_failure` разбирается в Задаче 0.2 — не смешивать.)

## Задача 0.2 — Развести конфликт деградации (`_distribute` vs `handle_account_failure`)

**Контекст.** Инциденты #2 и #3: выживший аккаунт забирал всю нагрузку и тоже улетал в бан. В коде **два противоречащих механизма**:
- `_distribute()` (poller.py:515) — правильно НЕ отдаёт каналы недоступному аккаунту.
- `pool.handle_account_failure()` (pool.py:143) — НЕПРАВИЛЬНО перекидывает каналы упавшего аккаунта на выжившего (`target = min(healthy, key=channel_count)`), вызывается из `health_check_loop` (pool.py:177). Именно это добивало второй аккаунт.

Это главная правка задачи: убрать перекидывание, а не «построить деградацию с нуля».

**Что сделать.**
1. Убрать/переписать `handle_account_failure()` так, чтобы каналы упавшего аккаунта НЕ перераспределялись на живые. При падении аккаунта — просто пометить unhealthy и оставить `_distribute` делать свою работу (он уже исключает недоступных). Проверить, что `_channel_assignments` больше не используется для форсированной переброски.
2. При `1 живой аккаунт`: Warm/Cold/Dormant ставятся на паузу, остаётся Hot с интервалом ×2 (`_get_effective_interval` уже даёт ×2; пауза тиров — добавить проверку числа живых аккаунтов в `_run_tier_loop`).
3. При `все аккаунты в бане`: поллинг стоп, алерт (Задача 1.4), ждём истечения CB.
4. После восстановления — пониженный бюджет 48ч (см. Задача 2.2).

**Приёмка.** Тест: падение Account 2 НЕ приводит к росту нагрузки на Account 1 (его каналы не перекидываются). При 1 живом аккаунте Warm/Cold/Dormant не опрашиваются, суммарная нагрузка ≤ бюджета одного аккаунта. Ручная трассировка `health_check_loop` при падении — нет вызова переброски.

## Задача 0.3 — Отключить поллинг Dormant для стран без подписчиков

**Контекст.** 1827 Dormant-каналов жгут бюджет и риск ради нулевой пользы — там нет подписчиков.

**Что сделать.**
1. В билдере тиров: канал попадает в расписание поллинга **только** если его страна имеет хотя бы одну активную подписку (`user_subscriptions` → `country_id`).
2. Каналы стран без подписчиков → статус `parked` (не поллятся). Хранить флаг в БД или вычислять на лету при ребилде тиров.
3. Конфиг-флаг `POLL_PARKED_COUNTRIES=false` (по умолчанию выключено).
4. Механизм «активации страны» реализуется в Задаче 1.5 — здесь только исключаем из расписания.

**Приёмка.** После ребилда тиров при текущих данных (подписки только по активным странам) в расписании остаётся 200–400 каналов, а не 2035. Лог показывает число `parked`.

## Задача 0.4 — Убрать параллельные батчи, ввести лог-нормальные паузы

**Контекст.** В `_poll_batch` (poller.py:384) каналы опрашиваются через `asyncio.gather` пачками по `PARALLEL_BATCH=3` (poller.py:33, 418-420) — три одновременных запроса с одного аккаунта, машинный паттерн. Jitter сейчас только на `BATCH_PAUSE` (poller.py:424, ±30% uniform) — недостаточно.

**Что сделать.**
1. В `_poll_batch` убрать `asyncio.gather` по каналам одного аккаунта — опрос строго **последовательный** (цикл по каналам с паузой между ними). `PARALLEL_BATCH` убрать или зафиксировать =1. Разные аккаунты по-прежнему работают параллельно (это уровень `_run_tier_loop`/`_distribute`, его не трогать).
2. Паузу между запросами брать из **лог-нормального распределения**, а не uniform: медиана ~2 сек, диапазон ~0.8–6 сек. Пример:
   ```python
   import random
   def next_delay() -> float:
       # медиана e^0.7 ≈ 2.0 c, тяжёлый правый хвост
       d = random.lognormvariate(0.7, 0.5)
       return min(max(d, 0.8), 6.0)
   ```
3. Порядок каналов внутри сессии перемешивать (`random.shuffle`) на каждом проходе.

**Приёмка.** В логах поллинга — последовательные запросы с одного аккаунта, интервалы распределены лог-нормально (не константа, не равномерно). Unit-тест на `next_delay()`: 1000 сэмплов в диапазоне [0.8, 6.0], медиана в районе 1.8–2.2.

## Задача 0.5 — Пер-аккаунтный rate limiter + суточный бюджет запросов

> **Делать ПЕРВОЙ.** Это первопричина, по которой warmup/jitter/staggered startup не спасали от банов. Бюджет поверх глобального лимитера бессмыслен — сначала лимитер должен стать пер-аккаунтным.

**Контекст.** `TelegramRateLimiter` (`app/userbot/rate_limiter.py`) — синглтон (`limiter = TelegramRateLimiter()`, строка 175) с **одним** `self._last_call` (строка 42) и одним `self._lock`. При двух аккаунтах `acquire()` ограничивает их **суммарный** темп: `DEFAULT_MIN_INTERVAL=0.3` = 3 rps на оба вместе, а не на каждого. Circuit breaker при этом уже пер-аккаунтный (ключи `circuit:open:{id}`) — а лимит нет. Это фундамент, на который наслаивались все anti-ban меры.

**Что сделать.**

Часть A — пер-аккаунтный лимитер:
1. Переделать `TelegramRateLimiter` так, чтобы `_last_call` и `_lock` были **на каждый account_id** (dict внутри, или отдельный инстанс на аккаунт в пуле). `acquire()` принимает `account_id`.
2. Обновить всех вызывающих: в poller.py `await limiter.acquire()` (строки 240, 330) должны передавать `account.account_id`.
3. `min_interval` вынести в конфиг, поднять до 1.0–2.0 сек на аккаунт (0.3 — агрессивно; см. рекомендацию аудита).

Часть B — суточный бюджет поверх:
4. Redis-счётчик `budget:used:{account_id}:{date}` (INT, TTL до конца суток).
5. Инкремент **перед каждым** API-вызовом аккаунта (внутри `acquire(account_id)` — тогда учёт автоматом покрывает все вызовы).
6. Конфиг `daily_request_budget` (по умолчанию 10000/аккаунт; карантинный множитель для нового — см. Задача 2.5).
7. При исчерпании бюджета аккаунт **останавливается** до следующих суток независимо от очереди. Лог + алерт (Задача 1.4).
8. Метод `budget_remaining(account_id) -> int` для планировщика.

**Приёмка.**
- Тест A: два аккаунта делают запросы — каждый ограничен своим `min_interval`, не делят один лимит (замерить, что параллельные вызовы разных аккаунтов не сериализуются в общую очередь).
- Тест B: при `daily_request_budget=100` 101-й запрос аккаунта блокируется, аккаунт помечен «бюджет исчерпан»; счётчик сбрасывается на смене суток; второй аккаунт со своим бюджетом не затронут.

## Задача 0.8 — Circuit breaker переживает рестарт контейнера

# ФАЗА 1 — Надёжность и наблюдаемость (неделя 1)

Цель: система сама сообщает о проблемах, а поллер работает «человекоподобно».

## Задача 1.1 — Сессионная модель планировщика

**Контекст.** Инцидент #3 — 12.5 часов ровного ритма без единой длинной паузы. Нужно имитировать человека с открытым Telegram.

**Что сделать.**
1. Планировщик каждого аккаунта работает **сессиями**, а не непрерывным циклом:
   - Активное окно: случайная длина 20–60 мин (`random.uniform`). Внутри — опрос каналов по расписанию тиров с паузами из Задачи 0.4.
   - Пауза между окнами: случайная 15–60 мин.
2. **Ночной сон:** 4–6 часов в сутки аккаунт полностью молчит. Час начала сна — свойство аккаунта (см. Задача 1.2).
3. Порядок обхода каналов внутри окна перемешивается.

**Приёмка.** Симуляция суток (ускоренное время в тесте): аккаунт имеет активные окна, паузы, один непрерывный «сон» 4–6 ч. Нет ни одного отрезка непрерывной активности > 60 мин.

## Задача 1.2 — Разные «таймзоны» аккаунтов

**Контекст.** Если оба аккаунта спят одновременно, покрытие Hot проваливается.

**Что сделать.**
1. Каждому аккаунту — свойство `sleep_start_hour` (час начала ночного сна), разнесённое между аккаунтами (напр. акк1 спит 02:00–07:00, акк2 — 14:00–19:00 по UTC).
2. Пока один аккаунт спит, второй держит Hot с интервалом ×2 (переиспользовать логику деградации из Задачи 0.2 / 1.3).
3. Хранить `sleep_start_hour` в конфиге аккаунтов или БД.

**Приёмка.** Симуляция: в любой момент суток хотя бы один аккаунт активен (при 2 живых). Провалов покрытия Hot нет.

## Задача 1.3 — Расширить адаптивный Hot-интервал (уже есть в зачатке)

**Контекст.** `_get_effective_interval` (poller.py:557) **уже** есть: 2+ аккаунта → base, 1 → base×2. Задача — расширить, а не создать. Также базовый `HOT_INTERVAL=60` (poller.py:29) несовместим с 2 аккаунтами (см. ключевую директиву) — поднять.

**Что сделать.**
1. Расширить `_get_effective_interval`: 3+ живых → нижняя граница (10–15 мин), 2 → 15–30 мин, 1 → удвоенное + пауза Warm/Cold/Dormant (Задача 0.2).
2. Поднять базовые интервалы: `HOT_INTERVAL` до 15–30 мин. Все значения — в конфиг, убрать хардкод из констант модуля.
3. Пересчёт при изменении числа живых аккаунтов (уже вызывается в цикле — проверить, что реагирует на бан/восстановление).

**Приёмка.** Тест: 3→2→1 живой аккаунт → интервал растёт по ступеням; при 1 Warm/Cold/Dormant на паузе. Значения читаются из конфига.

**Контекст.** Вопреки аудиту (8.13 «написан, выключен»), файла `llm_validator.py` в репозитории **нет**. В `config.py` есть только `deepseek_api_key` и `deepseek_model` — ни модуля, ни флага `DEEPSEEK_ENABLED`, ни вызова из classifier. Это greenfield-задача. **Приёмка.** На наборе известных false positives (EXASIA, Kymco, «Наличный обмен») LLM отсекает рекламу, пропускает реальный спрос. При выключенном флаге поведение = прежнее rule-based. При недоступности LLM система не молчит (fail-open).

## Задача 4.3 — Апгрейд VPS до 4 GB
**Триггер:** добавление 3-го аккаунта (+150–200 MB на Telethon-клиент) + включение LLM-валидации.
Пересмотреть memory limits контейнеров под новый объём.
**Приёмка.** Под пиковой нагрузкой (3 аккаунта + LLM + буфер очереди) нет OOM, swap не активен постоянно.

## Задача 4.4 — Авто-правила из фидбека + few-shot в LLM
**Триггер:** сотни записей в `feedback`.
На основе накопленного фидбека: авто-предложение стоп-слов (с ручным подтверждением в админке), few-shot примеры в промпт LLM-валидатора, A/B правил классификации.
**Приёмка.** Предложения стоп-слов формируются из фидбека и требуют подтверждения; few-shot улучшает precision на held-out наборе.

## Задача 4.5 — Расширенный мониторинг (Grafana/Prometheus)
**Триггер:** Telegram-алертов (1.4) + Sentry перестаёт хватать.
Prometheus-метрики (бюджет, интервалы, матчи, очередь, баны) + Grafana-дашборды. До этого триггера — не трогать, Sentry + Telegram покрывают.
**Приёмка.** Дашборды показывают бюджет per-account, историю банов, длину очереди, rate матчей.
